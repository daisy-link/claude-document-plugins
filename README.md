# 🎉 lo-cal Claude Code スキル・マーケットプレイス

**「仕様書を作って、仕様書どおりに作る」** を Claude Code で実現する、lo-cal 社内向けのプラグイン配布リポジトリです。

このリポジトリ自体がマーケットプレイス（`lo-cal-skills`）になっていて、プラグインをインストールするだけで全スキルが使えるようになります✨

---

## 🤔 このプラグインでできること

`spec-tools` プラグインひとつで、**設計 → 実装 → レビュー → テスト → セキュリティ／パフォーマンスチェック**まで、開発の一連の流れを Claude におまかせできます。

### ✨ 特徴

- 📚 **仕様書ファースト** — コードより先に仕様書！`docs/` の仕様書を「正」として開発を進めるので、仕様と実装がズレません
- 🗣 **話しかけるだけ** — 「仕様書を作って」「実装して」と頼むだけで、適切なスキルが自動で動きます
- 🔍 **実装したら必ずレビュー** — 実装後は仕様書との整合性レビューが自動で走ります。レビュー忘れゼロ！
- 🔁 **テストが通るまで自動修正** — NGでも3回までは自動でリトライ。ただし無限には繰り返しません（ちゃんと上限があります）
- 🛡 **危険なコマンドは強制ブロック** — プロジェクト外への `rm` や `sudo` などは、実行される**前に**フックが自動で止めます
- 📋 **進捗が見える** — どの機能がどこまで進んだか、`docs/実装状況.md` に自動で記録されます

## 🚀 推奨の使い方（開発の流れ）

### STEP 1️⃣ まずは設計から 📝

対象プロジェクトで Claude Code を起動して、こう話しかけます：

> 「**仕様書を作って**」

既存コードがあれば分析して、まだ何も無ければ**対話形式のヒアリング**で、要件定義書・画面設計書・DB設計・詳細設計などを `docs/` に生成します。

### STEP 2️⃣ 仕様書どおりに開発 🔨

> 「**ログイン画面を実装からテストまでやって**」

すると自動でこの流れが回ります：

```
実装 → lint・型チェック・ビルド確認 → 仕様適合レビュー → テスト作成・実行 → 回帰確認
         └─ NGなら自動で修正してリトライ（上限あり）─┘
```

### STEP 3️⃣ リリース前の品質チェック 🔒⚡

> 「**脆弱性がないかチェックして**」「**パフォーマンスを確認して**」

仕様書の認証・認可要件との突き合わせや、N+1・SQL効率・キャッシュ戦略の確認を行い、指摘一覧を提示。**承認した指摘だけ**を修正してテストまで回します。

### STEP 4️⃣ 変更したら仕様書も更新 🔄

> 「**仕様書を更新して**」

コードの変更点を検出して、関係する仕様書だけをきれいに更新します。

### STEP 5️⃣ 納品前にクライアント向けのテスト項目書も 📋

> 「**受け入れテスト項目書を作って**」

画面設計書・要件定義書をもとに、クライアントが実際に操作して確認できる平易な言葉のチェックリストを作成。文末には検収の署名欄も付きます。開発前（設計書だけの段階）でも作成でき、機能が増えたら「更新して」で差分だけ追記（記入済みの結果は上書きしません）。「**Excelでもほしい**」と言えば、会員関連・注文関連…のように業務ドメインごとにシートを分けたExcelファイルも追加で作れます。

### 🏃 仕様書なしでサクッと開発したいときは？

使い捨てスクリプトなど、仕様書が大げさな場面もありますよね。そんなときは：

> 「**仕様書なしで**実装して」

と**明示**すればOK！仕様適合レビュー等の品質保証が付かない旨を確認したうえで、通常の Claude Code として開発します（危険コマンドのブロックだけは常時有効です🛡）。

## 💡 こんなときに使えます（ユースケース）

| こんなとき 🙋 | 話しかけ方 💬 |
|--------------|--------------|
| これから新しいシステムを作る（まだコードが無い） | 「開発前だけど要件定義から設計書を作りたい」 |
| 既存プロジェクトにドキュメントが無い… | 「このリポジトリの仕様書を作って」 |
| 仕様書どおりに新機能を追加したい | 「商品管理APIを実装からテストまでやって」 |
| テストが無い機能にテストを足したい | 「注文機能のテストを作って」 |
| リリース前に安全確認したい | 「セキュリティチェックして」「N+1が無いか見て」 |
| コードを変更したので仕様書を追従させたい | 「決済APIを変更したから仕様書を更新して」 |
| 納品前にクライアントへ渡すテスト項目書がほしい | 「受け入れテスト項目書を作って」 |
| 仕様書とシステムの実態がズレていないか確認したい | 「仕様書とコードがズレていないか確認して」 |

## 🧰 スキル一覧

| スキル | なにをする？ |
|--------|-------------|
| 🏗 **architecture-design** | リポジトリを分析して（開発前なら対話で）、要件定義書・システム構成書・画面設計書・DB設計・詳細設計などのMarkdown仕様書を `docs/` に作成・更新・再作成 |
| 🎯 **spec-dev-cycle** | 開発作業の**唯一の窓口**！依頼を聞いて、下の5つの内部スキルをどの順序で使うか判断・実行するオーケストレーター |
| 🔨 spec-implementation | （内部）仕様書＋モック（任意）をもとにコードを実装。実装後のレビューまでセットで実行 |
| 🔍 spec-review | （内部）実装が仕様書・コーディング規約とズレていないかレビュー |
| 🧪 spec-test | （内部）仕様書から正常系・異常系・境界値のテストケースを洗い出してテストコードを作成 |
| 🛡 spec-security | （内部）仕様書の認証・認可要件と実装を突き合わせ、IDOR・インジェクション等の脆弱性をチェック |
| ⚡ spec-performance | （内部）N+1クエリ・SQL効率・キャッシュ戦略をDB設計・非機能要件と照らして確認 |
| 📋 **acceptance-test-writer** | 画面設計書・要件定義書をもとに、クライアント向けの受け入れテスト項目書（UAT資料）を作成。検収署名欄つき。開発前でも作成でき、希望すればドメイン別シート分けのExcelも出力 |
| 🔎 **spec-alignment-audit** | プロジェクト全体で仕様書と実装コードを突き合わせ、「仕様書にあるが未実装」「実装にあるが仕様書に未記載」を1件ずつ確認しながら解決。仕様書・実装コードの両方が揃っているプロジェクト向け |

> 💡 「（内部）」のスキルは直接呼び出す必要はありません。**普通に話しかければ spec-dev-cycle がぜんぶ交通整理してくれます**。開発系スキルはいずれも `docs/` の仕様書が前提なので、無い場合は先に仕様書作成を案内します。`acceptance-test-writer` と `spec-alignment-audit` は独立スキルなので直接呼び出せます（前者は `spec-dev-cycle` の開発サイクル完了時にも提案されます）。

## ⚠️ 制約事項（安全のためのルール）

安心して任せられるように、プラグイン全体に共通ルールを設けています。

### 🚫 禁止していること

- 🔐 **機密情報の書き込み禁止** — パスワード・APIキー・個人情報は、仕様書にもコードにも絶対に書き込みません
- 🐘 **システム全体の一括実装禁止** — 必ず「どの画面・機能から？」と確認し、機能単位で進めます
- ♾ **無限ループ禁止** — 修正の繰り返しは各ループ最大5回（自動は3回まで、4回目は確認、サイクル全体で合計10回）。上限に達したら潔く中止して報告します
- 👻 **常駐プロセス禁止** — watchモードのテストや開発サーバーは起動しません（テストはワンショット実行）
- 💣 **破壊的コマンド禁止** — プロジェクト外への `rm`・`mv`、`sudo`・`dd`・`mkfs` などは、**同梱フックが実行前に強制ブロック**します。指示ではなく仕組みで守ります💪

### 🤖 自動でやってくれること

- ✅ 実装後の**仕様適合レビュー**（レビューNGなら3回まで自動修正）
- ✅ **lint・型チェック・ビルド**の確認（プロジェクトに設定がある場合）
- ✅ 新規テストに加えて**既存テストも実行して回帰確認**
- ✅ 進捗を `docs/実装状況.md` に記録
- ✅ 仕様が不明な箇所は**着手前にまとめて質問** → 回答を仕様書に反映してから続行
- ✅ テスト失敗の原因が**環境問題**（DB未起動など）なら、無駄に修正せず即報告
- ✅ git 未導入のプロジェクトでは **`git init` を提案**（git なしを選んだ場合は変更前にバックアップを取って、中止時はそこから復元）
- ✅ 機能の区切りで**コミットを提案**（必ず承認制。push はあなたの手で🙌）

## 📦 導入手順

### 🖥 CLI（ターミナル）の場合

1. ターミナルで対象プロジェクトのフォルダに移動して、Claude Code を起動します

   ```bash
   cd あなたのプロジェクト
   claude
   ```

2. チャットに以下を入力して、マーケットプレイスを登録します

   ```
   /plugin marketplace add <このリポジトリのGit URL>
   ```

3. 続けてプラグインをインストールします

   ```
   /plugin install spec-tools@lo-cal-skills
   ```

4. **Claude Code を再起動**します（破壊的コマンドをブロックするフックは再起動後に有効になります🛡）

これで完了！あとは「仕様書を作って」と話しかけるだけです🎉

### 🖱 デスクトップ版（Mac / Windows）の場合

1. Claude Code デスクトップアプリで対象プロジェクトのフォルダを開きます
2. チャット欄に CLI と同じコマンド（`/plugin marketplace add ...` → `/plugin install spec-tools@lo-cal-skills`）を入力します
3. アプリを再起動すれば準備OK！

> 🧪 ローカルで試したい場合は、Git URL の代わりにこのリポジトリのパスも指定できます：
> `/plugin marketplace add /path/to/claude-skills`

### 🔄 更新について（定期的にチェックしてね！）

このプラグインは**どんどん改善・機能追加**しています🚀
`/plugin` を開いて `spec-tools` に更新が来ていないか、**定期的にチェック**してください。更新したら Claude Code の再起動もお忘れなく！

## 🔧 開発・メンテナンス（このリポジトリを触る人向け）

```
claude-skills/                          # = マーケットプレイス lo-cal-skills
├── .claude-plugin/
│   └── marketplace.json                # 配布カタログ
└── plugins/
    └── spec-tools/                     # プラグイン
        ├── .claude-plugin/plugin.json  # プラグイン定義（配布時に version を更新）
        ├── hooks/
        │   ├── hooks.json              # PreToolUseフック定義
        │   └── check_destructive.py    # 破壊的コマンドの検査スクリプト
        └── skills/
            ├── 共通ルール.md            # 全スキル共通の絶対禁止事項・トークン節約方針
            ├── architecture-design/    # SKILL.md + references/
            ├── spec-dev-cycle/         # SKILL.md（唯一の窓口）
            ├── spec-implementation/    # SKILL.md（内部スキル）
            ├── spec-review/            # SKILL.md（内部スキル）
            ├── spec-test/              # SKILL.md（内部スキル）
            ├── spec-security/          # SKILL.md（内部スキル）
            ├── spec-performance/       # SKILL.md（内部スキル）
            ├── acceptance-test-writer/ # SKILL.md（独立スキル）
            └── spec-alignment-audit/   # SKILL.md（独立スキル）
```

- スキルの追加・修正は `plugins/spec-tools/skills/` 配下で行う
- 全スキル共通のルールは `skills/共通ルール.md` に集約されている（各スキルから参照）
- 新しいプラグインを追加したら `.claude-plugin/marketplace.json` の `plugins` 配列に登録する
- 変更を配布する際は `plugin.json` の `version` を更新する。利用者は `/plugin` から更新できる
