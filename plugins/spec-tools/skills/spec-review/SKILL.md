# 仕様適合レビュー・スキル

`docs/` の仕様書とコーディング規約を正として、実装コードがそれらに違反していないかをレビューする。
一般的なロジックバグの検出・可読性の簡略化提案は組み込みの `code-review` / `simplify` スキルに譲り、
本スキルは**「仕様書に書かれたことが実装されているか」「規約から外れていないか」**に絞って見る。

> [!IMPORTANT]
> このスキルは**単体でユーザーから直接呼び出さない**。レビューに関する依頼は必ず
> `../spec-dev-cycle/SKILL.md` が窓口となり、そこから本ファイルが読み込まれて実行される
> （spec-implementation の実装後レビュー、spec-dev-cycle の単一フェーズ実行のいずれからも
> 同じ手順を使う）。

> [!CAUTION]
> `../共通ルール.md` に spec-tools プラグイン全体の絶対禁止事項とトークン節約方針がある。
> 修正を依頼された場合、機密情報・個人情報を書き込まない。修正の繰り返しが必要な場合は
> 最大5回までとする。

---

## STEP 1: 前提確認（必須）

```bash
find docs/ -name "*.md" 2>/dev/null | sort
test -f docs/コーディング規約.md && echo "コーディング規約あり"
```

- `docs/` に仕様書が見つからない場合 → レビューを開始せず、次のように案内して停止する：

  > 「`docs/` に仕様書が見つかりません。仕様適合レビューには仕様書が必要です。
  > 先に architecture-design スキルで仕様書を作成しますか？」

- `docs/コーディング規約.md` が無い場合は、その観点（STEP 3）のみスキップし、
  仕様書との整合性チェック（STEP 2）は通常どおり実施する。

---

## STEP 2: レビュー対象の特定

```bash
git status --short 2>/dev/null
git diff --stat 2>/dev/null
```

- ユーザーが対象ファイル・機能を指定した場合はそれに従う。
- 指定が無い場合、Git管理下なら直近の差分（未コミット変更、または直近コミット）を対象にする。
- Git管理外、またはリポジトリ全体を対象にする依頼の場合は、`docs/詳細設計/` や
  `docs/画面設計書/` の項目単位でファイルを紐付けてレビューする。

対象コードに**直接関連する**仕様書（該当する詳細設計・画面設計書・DB設計・要件定義書の該当箇所）
だけを読み込む。`docs/` 配下を無差別に全件読み込まない（`../共通ルール.md` のトークン節約方針）。

---

## STEP 3: 仕様書との整合性チェック

仕様書の記載と実装コードを1項目ずつ突き合わせる。観点は仕様書の種類ごとに異なる。

| 仕様書 | チェック観点 |
|--------|------------|
| 要件定義書 | 実装が要件でカバーされているか。要件に無い機能が紛れ込んでいないか |
| 画面設計書 | 画面項目・入力チェック・遷移先が仕様どおりか。項目の過不足 |
| DB設計 | テーブル・カラム名、型、制約（NOT NULL・UNIQUE・外部キー等）が定義どおりか |
| 詳細設計（API等） | エンドポイント・パラメータ・レスポンス構造・エラーコードが定義どおりか |
| システム設定仕様書 | 設定値のハードコードがないか。設定項目が仕様どおり読み込まれているか |

**「仕様書に無いことをコードが勝手にやっている」ケースも違反として扱う**（仕様の過不足は
どちらの方向も指摘する）。

---

## STEP 4: コーディング規約チェック（規約がある場合のみ）

`docs/コーディング規約.md` に明記されたパターン（バリデーションの置き場所・DB接続の方法・
ロジックの配置ルール・命名規則等）から外れている箇所を洗い出す。

---

## STEP 5: 結果の報告

重要度別に整理して報告する。重要度の目安：

| 重要度 | 内容 |
|--------|------|
| 🔴 違反 | 仕様書の記載と実装が明確に食い違っている（バグの温床・仕様外動作） |
| 🟡 要確認 | 仕様書の記載が曖昧/未記載で、実装が妥当か判断がつかない箇所 |
| 🔵 規約違反 | コーディング規約から外れているが動作上は問題ない箇所 |

```
【仕様適合レビュー結果】
対象: src/routes/items.ts（仕様書: docs/詳細設計/API_商品管理.md）

🔴 違反 (1件)
- GET /api/items のレスポンスに仕様書で定義された `stock` フィールドが含まれていない
  → items.ts:42 で SELECT 時に stock カラムを取得していない

🟡 要確認 (1件)
- 在庫0件時の挙動が仕様書に未記載。現状は空配列を返しているが仕様として明記が必要
  → 仕様書側の追記を推奨

🔵 規約違反 (1件)
- バリデーションがルーターに直書きされている（規約ではservice層に置く指定）
  → items.ts:15-20

違反なしの場合:「仕様書・コーディング規約との整合性チェック完了。違反はありませんでした。」
```

- 修正は自動で行わず、まず結果を報告する。ユーザーが修正を依頼した場合は該当箇所を修正する。
- 🟡要確認（仕様書側の不備・曖昧・未記載）は上記のとおり報告に含めるだけでよい。ただし、
  仕様が不明で「違反かどうかの判断自体ができない」場合は、報告を待たず
  `../spec-dev-cycle/SKILL.md` の「仕様不明・コーディング規約不在時の共通確認フロー」に従って
  その場でユーザーに確認する（本スキル自身は `docs/` を書き換えない。反映は architecture-design
  経由で行う）。

## 品質チェックリスト

- [ ] レビュー開始前に `docs/` の仕様書の有無を確認し、無ければ実施せず案内している
- [ ] 一般的なロジックバグ・簡略化提案には踏み込まず、仕様適合と規約遵守に絞っている
- [ ] 「仕様に無い実装」「実装に無い仕様」の両方向をチェックしている
- [ ] コーディング規約が無い場合は規約チェックのみスキップし、仕様整合性チェックは実施している
- [ ] 重要度（違反/要確認/規約違反）を分けて報告している
- [ ] 仕様書側の不備は仕様書修正の提案として伝え、勝手に `docs/` を書き換えていない
- [ ] 対象に直接関連する仕様書だけを読み込んでいる（`docs/` を無差別に全件読み込んでいない）
- [ ] 修正を行う場合、生成・編集したコードに機密情報・個人情報を書き込んでいない
