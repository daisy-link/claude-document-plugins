# テスト項目書作成スキル

対象プロジェクトの設計書（基本設計書・詳細設計書など）を読み込み、画面操作・API単位の
結合テスト・システムテストの観点でテスト項目書をMarkdown形式で作成する。

> [!NOTE]
> このスキルは `system-spec-writer` / `spec-to-html` と組み合わせて使う姉妹スキル。
> `system-spec-writer` が `docs/` に生成した仕様書を入力として使う。
> `docs/` に仕様書がまだ無い場合は、先に `system-spec-writer` で仕様書を作ることを提案する。

---

## 対象とするテストレベル

**結合テスト・システムテストレベル**（画面操作単位 / API単位の機能テスト）を対象にする。
関数・メソッド単位の単体テストは対象外。ユーザーが単体テストレベルを求めている場合は、
粒度が異なる旨を伝えて方針を確認する。

---

## 出力ファイルの構成

出力先は `docs/テスト項目/` フォルダ。ファイルは**実行日の日付で `YYYYMMDD.md`**として保存する（例：`20260702.md`）。
「1つの仕様変更 = 1ファイル」ではなく、**「その日に追加したテストケースをまとめたファイル」**という単位で扱う。
同じ日に複数回テストケースを追加することになった場合は、その日のファイルに追記していく。

- `docs/index.md` へのリンクは不要（このフォルダは一覧・HTML化の導線に含めない）
- 過去の日付のファイルは**読むだけで、内容には一切手を加えない**（実施結果を保護するため）

## 動作モードの判定（最初に確認）

```bash
ls -1 docs/テスト項目/*.md 2>/dev/null
date +%Y%m%d
```

| 判定条件 | モード |
|---------|--------|
| `docs/テスト項目/` が無い、または今日の日付のファイルが無い | **新規ファイル作成** → 今日の日付で新しいファイルを作る |
| 今日の日付のファイル（`docs/テスト項目/YYYYMMDD.md`）がすでにある | **追記** → そのファイルの末尾に追加する |

### 【絶対ルール】

テスト項目書は「仕様が作成・更新されるたびに、必要なテストケースを積み増していく」台帳として扱う。
**過去の日付のファイルおよびそこに記入された結果・備考は一切書き換えない・削除しない。**
仕様の変更に対応するテストが必要になったら、過去のファイルを編集するのではなく**今日の日付のファイルに新しいテストケースとして追加する**。

> [!IMPORTANT]
> 「テスト項目書を更新して」と言われても、これは「既存のテストケースを最新の仕様に書き換える」という意味ではなく
> 「今回の仕様変更に対応するテストケースを追加する」という意味で解釈する。過去のファイルには触れない。

#### 0. 旧形式（単一の `docs/テスト項目書.md` ファイル）からの移行

`docs/テスト項目/` フォルダが無く、代わりに古い単一ファイル `docs/テスト項目書.md` が見つかった場合、
**この移行のときだけ**例外的に既存内容に手を加えてよい。

1. `docs/テスト項目/` フォルダを作り、旧ファイルの内容を今日の日付のファイルとして保存し直す
2. 列構成が STEP 3 の形式（No/大項目/小項目/手順/期待結果/結果/備考）と異なる場合は合わせて整える。
   「OK」「○」「済」など合格を示す記載があれば結果列を `[x]`、空欄・「未実施」・「NG」などは `[ ]` にし、
   NGの理由や既存の備考はそのまま備考欄に引き継ぐ
3. 元の `docs/テスト項目書.md` を削除してよいかユーザーに確認してから削除する
4. 移行した旨をユーザーに報告する

#### 通常の追加

1. 今日の日付のファイル（`docs/テスト項目/YYYYMMDD.md`）が既にあれば読み込み、既存の No・内容を把握する（**この時点の内容には一切手を加えない**）。無ければ新規作成として扱う
2. 何に対するテストケースが新しく必要かを把握する
   - `system-spec-writer` の STEP 6 から呼ばれた場合：直前に作成・更新した仕様書の変更箇所をそのまま対象にする
   - 単独で「追加して」と頼まれた場合：`git status`/`git diff` や仕様書の変更点、ユーザーの説明から対象を特定する
3. 対象範囲について STEP 2（観点抽出）〜STEP 3（作成）の要領で新しいテストケースだけを作る
4. 今日のファイルの一覧表の末尾に新しい行を追加する。No はそのファイル内での続き番号にする（新規ファイルなら1から、追記なら既存の最大値の続きから）。結果列は `[ ]`（未実施）にする
5. ファイル冒頭に「元になった仕様変更」を一言メモしておく（複数回追記した場合は追記のたびに一行足す）

```markdown
## 追加履歴

| 追加内容 |
|---------|
| 〇〇API追加に伴うテストケース（No.5〜8） |
| 初回作成（△△画面まわり） |
```

## STEP 1: 入力の仕様書を確認する

```bash
# 仕様書フォルダの有無を確認
ls -1 docs/*.md 2>/dev/null
```

主に以下を読み込む（存在するものだけでよい）。

| 参照する仕様書 | テスト項目書への使い道 |
|--------------|----------------------|
| 基本設計書 | 画面一覧・画面操作・システム構成 → 画面単位のテスト大項目 |
| 詳細設計書 | API仕様・DBテーブル定義・入出力項目 → API単位のテスト大項目、入力値の境界値 |
| 要件定義書 | 機能の重要度・制約事項 → 重要な機能のテストケース洗い出し漏れ防止 |
| システム設定仕様書 | 設定値による挙動差異 → 設定変更時のテスト観点 |

`docs/` が存在しない、または仕様書が無い場合は、ソースコード（画面コンポーネント・APIルーティング・DBスキーマ）を直接読み込んで代替する。
その場合は仕様書が無いことをユーザーに伝え、`system-spec-writer` で先に仕様書を作る選択肢も案内する。

---

## STEP 2: テスト観点の抽出

画面・APIごとに、以下の観点で洗い出す。**正常系だけで終わらせず、必ず異常系・境界値も検討する。**

| 観点 | 内容 | 例 |
|------|------|------|
| 正常系 | 想定通りの入力・操作での動作確認 | 必須項目を全て正しく入力して登録 → 成功する |
| 異常系 | 不正な入力・操作での動作確認 | 必須項目未入力で送信 → エラーメッセージが表示される |
| 境界値 | 最大/最小値、文字数上限、0件/1件/上限件数 | 文字数上限ちょうど／上限+1文字での挙動 |
| 権限・認証 | ログイン状態・権限による制御 | 未ログインでアクセス → ログイン画面にリダイレクト |
| データ整合性 | DB制約（一意制約・外部キー・NOT NULL）の検証 | 既存の一意キーで登録 → 重複エラー |

> [!TIP]
> 詳細設計書に入力項目の型・桁数・必須/任意が書かれていれば、それがそのまま境界値・異常系のテストケースの元になる。

---

## STEP 3: テスト項目書の作成

テスト項目書は **1本の一覧表**にする。列は少なくシンプルに保ち、結果もその場で記入できるようにする。

| No | 大項目 | 小項目 | 手順 | 期待結果 | 結果 | 備考 |
|----|--------|--------|------|---------|------|------|

- **大項目**: 画面名 または API名（例：「ユーザー登録画面」「POST /api/users」）
- **小項目**: 具体的なテストの内容（例：「メールアドレス未入力で送信」）。STEP 2の観点（正常系/異常系/境界値/権限・認証/データ整合性）を意識して洗い出すが、列としては持たない
- **手順**: 前提条件を含めて、実際に行う操作をひとつづきの文章にする（例：「登録画面を表示し、メールアドレスを空欄のまま他項目を入力して送信する」）
- **期待結果**: 具体的に書く（あいまいな表現にしない）
- **結果**: `[ ]` = 未実施 または NG、`[x]` = OK。**新規作成時はすべて `[ ]`**
- **備考**: NGの内容、要確認事項、補足など。実施日・実施者を書き添えたい場合もここに書く

> [!NOTE]
> 表のセル内の `[ ]` / `[x]` は、GitHubの箇条書き（`- [ ]`）のような**クリックで切り替わる本物のチェックボックスにはならない**（表のセル内では文字としてしか扱われない）。
> 実施したら**セル内のスペースを `x` に書き換える**ことでOK扱いにする運用にする。

### 記載例（`docs/テスト項目/20260702.md`）

```markdown
# テスト項目書（2026-07-02 追加分）

## 追加履歴

| 追加内容 |
|---------|
| ユーザー登録画面・ユーザー登録APIの初回作成 |

## テスト項目一覧

| No | 大項目 | 小項目 | 手順 | 期待結果 | 結果 | 備考 |
|----|--------|--------|------|---------|------|------|
| 1 | ユーザー登録画面 | メールアドレス未入力で送信（異常系） | 登録画面を表示し、メールアドレスを空欄のまま他項目を入力して送信する | 「メールアドレスを入力してください」と表示され登録されない | [ ] | |
| 2 | ユーザー登録画面 | 正常な値で登録（正常系） | 登録画面を表示し、全項目を正しい形式で入力して送信する | 登録が成功し完了画面に遷移する | [x] | 2026-07-10 田中 |
| 3 | POST /api/users | emailが既存ユーザーと重複（データ整合性） | 既に登録済みのemailと同じ値でリクエストを送る | 409エラーと重複エラーメッセージが返る | [ ] | タイムアウトが発生しNG。要調査 |
```

### テストデータに関する注意

テスト手順・前提条件に登場する値は、`system-spec-writer` と同じ考え方で**ダミーデータのみ**を使う。
実際の個人情報・本番の認証情報（パスワード・APIキー・実在のメールアドレス等）は絶対に記載しない。
`test@example.com` のような明らかなダミー値を使う。

## STEP 4: ファイルの保存と提示

出力先は `docs/テスト項目/YYYYMMDD.md`（実行日の日付、システム名やアンダースコアは付けない）。

```bash
mkdir -p docs/テスト項目
```

`docs/index.md` へのリンク追加は**不要**（このフォルダは一覧・HTML化の導線には含めない）。

生成が終わったら、以下を簡潔に報告する。**新規ファイル作成の場合：**

```
【テスト項目書 作成完了】
- 出力先: docs/テスト項目/20260702.md
- 件数: 32件（正常系12 / 異常系14 / 境界値4 / 権限2）
- 結果: すべて未実施（[ ]）
- [要確認] 事項: 決済APIのタイムアウト仕様が仕様書に無かったため、テストケースを保留にしています
```

**同じ日のファイルに追記した場合：**

```
【テスト項目書 追加完了】
- 出力先: docs/テスト項目/20260702.md（既存ファイルに追記）
- 追加したテストケース: 4件（No.33〜36）
- 追加理由: 〇〇API追加に伴う
- 同ファイル内の既存のテストケース・結果・備考: 変更なし（32件のまま）
```

## 品質チェックリスト

- [ ] 正常系だけでなく異常系・境界値・権限のテストケースが含まれている
- [ ] 各テストケースの期待結果が具体的（あいまいな記述がない）
- [ ] テストデータに実際の個人情報・本番認証情報を使っていない
- [ ] 仕様書から読み取れなかった箇所は `[要確認]` として報告している
- [ ] 出力先が `docs/テスト項目/YYYYMMDD.md`（実行日の日付）になっている
- [ ] 列構成が「No/大項目/小項目/手順/期待結果/結果/備考」になっている（新規作成時は結果がすべて `[ ]`）
- [ ] 過去の日付のファイルには一切手を加えていない
- [ ] 今日の日付のファイルに追記する場合、既存のテストケース・結果・備考を一切書き換えず、末尾に追加している
- [ ] `docs/テスト項目書.md`（旧形式の単一ファイル）からの移行が発生した場合、削除前にユーザーへ確認している
- [ ] `docs/index.md` へのリンクは追加していない（このフォルダは対象外でよい）

---

## 補足: system-spec-writer / spec-to-html との連携

- `system-spec-writer` で仕様書を作った直後なら、そのまま `docs/` を入力にしてこのスキルを使える。
- `docs/テスト項目/` は `system-spec-writer` の出力（`docs/直下の各仕様書.md`）とは別フォルダに置かれる。
  `spec-to-html` の変換スクリプトは入力ディレクトリ直下の `.md` しか見ないため、サブフォルダである
  `docs/テスト項目/` は自動的にHTML化・サイドバーの対象外になる（index.mdへのリンクが不要なのはこのため）。
