# 仕様書ベース開発サイクル・オーケストレーション・スキル

`docs/` の仕様書に基づく開発作業（実装・レビュー・テスト・セキュリティ・パフォーマンス）は、
**すべてこのスキルを窓口にする。** 5つの内部スキルをユーザーから直接トリガーすることはない
（各スキルの `SKILL.md` 側にもその旨を明記している）。単一フェーズだけの依頼でも、複数フェーズの
組み合わせ・繰り返し実行でも、まず本スキルが依頼内容を解釈し、どの内部スキルをどう使うかを判断する。

**各フェーズの詳細な作業手順（何を読み込み、どう実装/レビュー/テスト/チェックするか）は
このスキルでは重複定義しない。** 該当フェーズに入ったら、以下の各スキルの `SKILL.md` を読み込み、
その指示にそのまま従う。本スキルが扱うのは「どの内部スキルを使うか」「どの順番で」
「どこまで自動で繰り返すか」「いつユーザーに確認するか」というオーケストレーション部分だけ。

| フェーズ | 参照するスキル |
|---------|---------------|
| 実装（レビュー込み） | `../spec-implementation/SKILL.md`（内部で `../spec-review/SKILL.md` によるレビュー・修正ループを実施する） |
| レビュー単体（既存コードの整合性確認のみ） | `../spec-review/SKILL.md` |
| テスト | `../spec-test/SKILL.md` |
| セキュリティチェック | `../spec-security/SKILL.md` |
| パフォーマンスチェック | `../spec-performance/SKILL.md` |

> [!IMPORTANT]
> 前提は各内部スキルと同じく `docs/` の仕様書が必須。`docs/` に仕様書が無い場合は何も実行せず、
> 先に architecture-design スキルでの仕様書作成を案内する。

> [!CAUTION]
> `../共通ルール.md` に spec-tools プラグイン全体の絶対禁止事項（機密情報の書き込み禁止・
> 繰り返し処理の上限〔各ループ5回・サイクル全体10回〕・環境起因エラーのカウント除外・
> 常駐プロセスの起動禁止）とトークン節約方針がある。本スキルおよびすべての内部スキルは
> これに従う。STEP 4の繰り返しルールはこの共通ルールに基づく。

## STEP 1: 実行パターンの判定

ユーザーの依頼文言から、以下のどのパターンかを判定する。曖昧な場合はユーザーに確認する。

| パターン | 判定条件の例 | 実行内容 |
|---------|------------|---------|
| 0. 単一フェーズ | 「実装して」「レビューして」「テストを作って」「脆弱性がないかチェックして」「パフォーマンスを確認して」など、1フェーズのみの依頼 | 該当する内部スキル1つだけを呼び出す（実装を含む場合はSTEP 2の対象範囲確認を行う） |
| A. フルサイクル | 「実装からテストまで」「テストが通るまで作って」「一気にやって」 | 実装（レビュー込み） → テスト。NG時はSTEP 4の繰り返しルールに従う |
| B. セキュリティ／パフォーマンス改善サイクル | 「セキュリティの指摘を直して」「パフォーマンスを改善して」 | 該当チェック → 指摘一覧提示 → 承認された指摘のみ対応 → テスト。NG時はSTEP 4の繰り返しルールに従う |

> [!NOTE]
> ユーザーが「spec-implementation を使って」のように内部スキル名を直接指定した場合も、
> このスキルが窓口となって該当フェーズを実行する（内部スキルへ直接処理を委譲しない）。

### 仕様書なしを明示された場合（スキップ）

ユーザーが「仕様書なしで実装して」「仕様書は作らなくていいから直して」のように
**明示的に**仕様書なしを希望した場合は、仕様書作成を強要しない。次の警告を1回だけ伝えて
承諾を得たら、本スキルおよび内部スキルを使わず、**通常の Claude Code として**開発を進める：

```
仕様書なしで進めます。この場合、仕様適合レビュー・仕様書ベースのテスト作成など、
このプラグインの品質保証は適用されません（破壊的コマンドのブロックは引き続き有効です）。
このまま進めてよいですか？
```

- 承諾を得たら通常の開発として対応する
- **明示が無い場合（単に `docs/` に仕様書が無いだけの場合）はこのスキップを適用しない。**
  従来どおり architecture-design での仕様書作成を案内する
- 後から「やっぱり仕様書も作って」と言われたら、architecture-design で追いつき生成すればよい

## STEP 2: 実装対象範囲の確認（実装を伴う場合のみ）

パターン0（実装）・パターンAのように実装フェーズを含む場合、着手前に対象範囲を確認する。

- 「システム全体を実装して」「まるごと作って」のように対象が広範・不明確な依頼の場合、
  いきなり着手せず次のように確認する：

  ```
  システム全体を一度に実装するのは避けたいです。どの画面・機能から着手しますか？
  ```

- `docs/実装状況.md` が存在する場合は参照し、実装済み・未着手の機能を踏まえて対象を提案する。
- ユーザーがすでに特定の画面・API・機能単位を指定している場合は、上記の絞り込み確認は省略してよい。
- 対象範囲が確認できたら、その範囲を対象として `../spec-implementation/SKILL.md` を実行する。

対象範囲が決まったら、実装に着手する前にバージョン管理の状態を確認する：

```bash
git rev-parse --is-inside-work-tree 2>/dev/null || echo "gitなし"
git status --short 2>/dev/null
```

**git 管理下の場合**：対象機能と無関係な未コミット変更が既にある場合は、今回の変更と
混ざる旨をユーザーに伝え、そのまま進めてよいか確認する（コミット・退避の判断はユーザーに
委ねる）。

**git 未導入の場合**：実装に着手する前に、次のように git の導入を提案する：

```
このプロジェクトは git 管理されていません。開発を安全に進めるため git を導入しますか？
（a）git init する（推奨。中止時の復元・差分レビュー・機能単位のコミットが可能になります）
（b）git なしで進める（中止時の復元はバックアップからの手動復元になります）
```

- **（a）承認された場合**：`git init` を実行し、現状を初期コミットしてから実装に着手する
  （以降は git 管理下として扱う）
- **（b）git なしを選ばれた場合**：**gitなしモード**として以下のルールで進める
  - 実装で既存ファイルを変更・削除する前に、`mktemp -d /tmp/spec-tools-backup.XXXXXX` で
    作成したバックアップディレクトリへ変更前のファイルを相対パス構造を保ってコピーし、
    新規作成したファイルは一覧として記録する
  - 中止時に「実装前の状態に戻す」を選ばれたら、バックアップから復元し、新規作成ファイルを
    削除する
  - STEP 6 のコミット提案はスキップし、代わりに完了報告でバックアップ先のパスを伝え、
    git 導入を改めて提案してよい

> [!CAUTION]
> システム全体の一括実装は行わない。1回の実行で着手する範囲は、ユーザーが指定した
> 画面・機能・API単位にとどめる。

## STEP 3: パターンごとの実行

### パターン0: 単一フェーズ

ユーザーの依頼に対応する内部スキル1つだけを実行する。

| 依頼内容 | 実行するスキル |
|---------|---------------|
| 実装 | `../spec-implementation/SKILL.md`（STEP 2の対象範囲確認を先に行う。内部でレビューも自動的に行われる） |
| レビュー（既存コードの整合性確認） | `../spec-review/SKILL.md` |
| テスト作成 | `../spec-test/SKILL.md` |
| セキュリティチェック | `../spec-security/SKILL.md` |
| パフォーマンスチェック | `../spec-performance/SKILL.md` |

実行後はそのままSTEP 6の完了報告に進む（テストの自動実行やSTEP 4の繰り返しルールは、
実装フェーズを実行した場合でも単一フェーズ実行では適用しない。テストまで含めたい場合は
ユーザーにパターンAでの実行を提案してよい）。

### パターンA: フルサイクル（実装 → テスト）

1. STEP 2の対象範囲確認を行う
2. `../spec-implementation/SKILL.md` の手順に従って実装する（実装後レビューと🔴違反の修正ループが
   このフェーズに内蔵されているため、本スキルから改めて spec-review を呼び出す必要はない）
3. `../spec-test/SKILL.md` の手順に従ってテストケースを洗い出し、テストコードを作成・実行する。
   **新規テストに加えて既存のテストスイート（少なくとも関連するスイート）も実行し、
   今回の変更が既存機能を壊していないか（回帰）を確認する**
4. テスト結果（新規・既存いずれか）がNGの場合 → STEP 4「繰り返しルール」に従う。実装への
   差し戻しは `../spec-implementation/SKILL.md` の手順で行うため、修正のたびに実装後レビューも
   自動的に再実行される
5. すべてのテストがOKの場合 → STEP 6の完了報告に進む。報告時に、セキュリティ／
   パフォーマンスチェック（パターンB）の実行を提案してよい（リリース前ゲート）

### パターンB: セキュリティ／パフォーマンス改善サイクル

1. 依頼内容に応じて `../spec-security/SKILL.md` または `../spec-performance/SKILL.md` の手順で
   チェックを実施し、指摘事項の一覧を作成する
2. 指摘事項をユーザーに提示し、**どの指摘に対応するか承認を得る**（全件対応／一部のみ対応を選べる）

   ```
   【対応確認】
   🔴 Critical: 注文APIの所有者チェック漏れ（IDOR）
   🟡 Medium: パスワードリセットトークンに有効期限が未設定

   これらに対応してよいですか？（全部/一部を選択/対応しない）
   ```

3. 承認された指摘のみ修正する（承認されなかった指摘は対応せず、報告にのみ残す）
4. 修正後、`../spec-test/SKILL.md` の手順で関連テストを実行する（既存テストがあれば実行し、
   関連するテストが無ければ最小限のテストケースを作成して実行する）。修正が広範囲に及ぶ場合は
   既存テストスイート全体で回帰確認する
5. テスト結果がNGの場合 → STEP 4「繰り返しルール」に従う
6. テスト結果がOKの場合 → STEP 6の完了報告に進む

---

## STEP 4: 繰り返しルール（無限ループ防止・最大5回）

テストがNGだった場合の再修正は、以下のルールで制御する。**このルールはパターンA・Bで共通。**
（`../spec-implementation/SKILL.md` 内部のレビュー・修正ループとは別に、テストの成否に対して
このルールを独立してカウントする。）**テスト実行は最大5回までとし、それ以上は繰り返さない**
（`../共通ルール.md` の絶対禁止事項「繰り返し処理は最大5回まで」）。

さらに以下を併用する（いずれも `../共通ルール.md` に基づく）：

- **テストはワンショット実行する**：watch モードで実行しない（`vitest run`・
  `jest --watchAll=false` 等、1回で終了する形式を使う）。タイムアウトを設定し、
  ハングした場合は打ち切って環境起因のエラーとして扱う
- **環境起因のNGはカウントしない**：失敗の原因が依存パッケージ未インストール・DB未起動・
  設定不足など環境の問題である場合、コード修正では解決しないためこのループを回さない。
  その場で処理を止め、環境の問題と解消方法をユーザーに報告する（環境が整ってから再開する）
- **サイクル全体の修正回数は合計10回まで**：このループの修正と、修正のたびに再入する
  spec-implementation 内部のレビュー・修正ループの修正を合算して10回に達したら、
  テスト回数が5回未満でも処理を中止して報告する

1. **1回目〜3回目のNGは自動で修正する（確認不要）**
   テストの失敗内容（と、あればレビューの指摘）をもとに実装を修正し、再度テストを実行する。
   3回目のNGまでは、確認を挟まず自動で修正・再テストを繰り返してよい。
2. **4回目のNGは、ユーザーに確認する**
   3回自動修正してもなおNGの場合は、そこで止めて状況を報告し、次のように確認する：

   ```
   【テスト結果】4回目もNGでした（失敗: 2件）
   - test_stock_negative: 在庫マイナス時のエラーメッセージが仕様と不一致
   - test_bulk_update: 一括更新で1件だけ反映されない

   このまま自動でもう1回修正しますか？それとも一旦ここで止めますか？
   ```

   - 「続けて」等の承認を得たら、**1回だけ**追加で修正・再テストする
   - 「中止」等の指示があれば、その時点で終了しSTEP 6の完了報告に進む
3. **5回目のテストでもNGの場合は、ユーザーの意向にかかわらずそこで処理を中止する**
   確認を挟まず「テストが5回連続でNGのため、これ以上は繰り返さず中止します」と報告し、
   残っている失敗内容と**これまでに試した修正アプローチ（何をどう直そうとしたか）**を明記して
   STEP 6の完了報告に進む。中止後にユーザーが同じ問題の修正を再依頼した場合は新しい
   サイクルとして扱ってよいが、前回試して失敗した修正アプローチをそのまま繰り返さない

理由を問わず修正の途中で中止した場合は、**書きかけの変更をそのまま残すか、実装前の状態に
戻すかをユーザーに確認する**（確認なしに変更を破棄しない）。「戻す」場合、git 管理下では
git の差分から復元し、gitなしモードではバックアップからの復元＋新規作成ファイルの削除で戻す。

> [!CAUTION]
> 4回目のNGに対する修正は、ユーザーの明示的な承認なしに自動実行しない。「多少失敗が残っても
> 一旦保存して報告する」ことを優先し、無限に修正し続けない。**テスト実行が5回に達したら、
> ユーザーが続行を希望しても打ち切る。**

## STEP 5: 仕様不明・コーディング規約不在時の共通確認フロー

実装・レビュー・テスト・セキュリティ・パフォーマンスいずれのフェーズを実行中でも、
以下のいずれかに該当し**自力で判断すると仕様を勝手に決めてしまうことになる場合**は、
そのフェーズの作業を中断し、このフローに従う。

- 仕様書に記載が無い、または曖昧で、実装・修正の方針が決められない
- `docs/コーディング規約.md` が存在せず、どの方針で実装するか決められない
- レビュー・テスト・セキュリティ・パフォーマンスチェックの過程で、判断の前提となる仕様が
  仕様書から読み取れない

> [!NOTE]
> 🟡要確認・軽微な曖昧さなど、各内部スキルが「報告するだけで作業を止めずに進められる」もの
> （例: spec-review の🟡要確認、spec-test/spec-security/spec-performance が最終報告に含める
> 改善提案）はこのフローの対象外。**その場で仕様を決めないと先に進めない場合だけ**このフローに従う。

### 5-0. 事前一括チェック（フェーズ着手前）

中断の繰り返しを防ぐため、各フェーズの作業に着手する**前**（対象範囲の仕様書を読み込む段階）で、
以下をまとめて洗い出す：

- 仕様書内の `[要確認]` 記載
- 対象範囲の作業に必要なのに記載が無い・曖昧な仕様
- `docs/コーディング規約.md` の有無（実装フェーズの場合）

不明点が1件でもあれば、着手前に5-1で**一括して**確認する。作業中に新たな不明点が見つかった
場合もこのフローを使うが、その時点で判明している不明点はまとめて1回で確認する
（1件ずつ何度も中断しない）。

### 5-1. ユーザーに確認する

不明点を具体的に整理してユーザーに確認する。複数あれば必ずまとめて確認する（1件ずつ分けて
確認しない）。

```
【確認が必要です】
- 在庫が0件のときのレスポンス仕様が仕様書に記載がありません。404を返しますか、空配列を返しますか？
- docs/コーディング規約.md が存在しません。どの方針で進めますか？
  （a）既存コードのパターンに合わせる（b）一般的なベストプラクティスに従う（c）この場で規約を指定する
```

> [!IMPORTANT]
> コーディング規約が無い場合に「既存コードから読み取る」「一般的なベストプラクティスに従う」を
> スキル側で自動選択してはいけない。必ずここでユーザーに方針を確認する。

### 5-2. 回答を architecture-design で仕様書に反映する

回答が得られたら、architecture-design スキルの手順を使って `docs/` の該当ファイルに反映する。

- 仕様の不明点 → 該当する仕様書（詳細設計・画面設計書・要件定義書等）に追記
- コーディング規約が無かった → ユーザーの回答をもとに `docs/コーディング規約.md` を新規作成する
  （「規約は作らずこの場限りの方針とする」という回答であれば、無理に作成せずその場限りの
  方針として扱ってよい）

### 5-3. 反映後、続行してよいか確認する

```
docs/詳細設計/API_商品管理.md に「在庫0件時は空配列を返す」を追記しました。
この内容で処理を続けてよいですか？
```

承認が得られたら、中断していたフェーズの作業を再開する。

## STEP 6: 完了報告

### 完了条件（Definition of Done）

実装を伴うサイクルは、以下をすべて満たしたときに「完了」とする：

- レビューの🔴違反が0件
- lint・型チェック・ビルドが通過している（プロジェクトに設定がある場合）
- 新規テストが全件PASSし、既存テストにも回帰が無い
- `docs/実装状況.md` を更新済み

満たせないまま中止した場合は、未達の項目を報告に明記する。

### 実装状況管理表の更新

実装を伴うサイクル（パターン0の実装・パターンA）を完了・中止したら、報告の前に
`docs/実装状況.md` を更新する（無ければ作成する）。画面・機能ごとの進捗を記録し、
次回のサイクルで「どこまで開発が進んでいるか」を参照できるようにする。

```markdown
# 実装状況

| 画面・機能 | 仕様書 | 実装 | レビュー | テスト | 最終更新 | 備考 |
|-----------|--------|------|---------|--------|---------|------|
| 商品管理API | 詳細設計/API_商品管理.md | ✅ | ✅ | ✅ 5件PASS | YYYY-MM-DD | |
| ログイン画面 | 画面設計書/P01_ログイン画面.md | ✅ | ✅ | ❌ 中止（2件NG） | YYYY-MM-DD | 試した修正アプローチは完了報告参照 |
| 注文管理API | 詳細設計/API_注文管理.md | 未着手 | - | - | - | |
```

> [!NOTE]
> `docs/実装状況.md` は仕様書ではなく進捗管理ファイルのため、本スキルが直接作成・更新してよい
> （architecture-design を経由しない。内部スキルの「`docs/` を書き換えない」制約の対象外）。

### 報告

サイクル全体の結果をまとめて報告する。

```
【開発サイクル完了】
仕様書: docs/詳細設計/API_商品管理.md

- 実装: src/routes/items.ts, src/db/migrations/xxx_add_stock.sql
  （実装フェーズ内蔵のレビューで🔴違反1件を1回目の自動修正で解消。🟡要確認1件は未対応）
- 静的チェック: lint・型チェック・ビルドすべて通過
- テスト: tests/routes/items.test.ts（5ケース）→ テスト再修正2回目で全件OK。既存テストにも回帰なし
- テスト繰り返し回数: 2回（いずれも自動修正）
- 仕様不明点: 在庫0件時の挙動をユーザーに確認し、docs/詳細設計/API_商品管理.md に反映済み
- 実装状況: docs/実装状況.md を更新（商品管理API: 実装✅ レビュー✅ テスト✅）
```

途中で中止した場合は、どこまで完了し何が残っているか、および**試した修正アプローチ**を
明確に報告する（再依頼時に同じ失敗を繰り返さないための記録）。

### 次のアクションの提案（任意）

報告の最後に、状況に応じて次を提案してよい：

- **コミットの提案**：機能単位の区切りとしてコミットを提案する。コミットは必ずユーザーの
  承認を得てから行い、push はユーザーに委ねる（git 管理下の場合のみ。gitなしモードでは
  スキップし、代わりにバックアップ先のパスを伝えて git 導入を改めて提案してよい）
- **リリース前ゲート**：フルサイクル（パターンA）完了後に「セキュリティ／パフォーマンス
  チェックも実行しますか？」と提案する（実行する場合はパターンBとして行う）
- **受け入れテスト項目書の提案**：フルサイクル（パターンA）またはセキュリティ／パフォーマンス
  改善サイクル（パターンB）が完了したタイミングで、「クライアント向けの受け入れテスト項目書
  （UAT資料）も作成しますか？」と一声かけてよい。承諾された場合は `../acceptance-test-writer/SKILL.md`
  を呼び出す（本スキル経由の内部スキルではなく独立スキルのため、そのまま丸ごと処理を委譲してよい）
