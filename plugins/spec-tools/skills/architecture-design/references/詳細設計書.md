# 詳細設計書の作り方

**画面を伴わない処理**（API・バッチ・RPA・CSV連携・外部サービス連携・共通処理）を、
種類ごと・項目ごとに個別ファイルでまとめる。

> [!IMPORTANT]
> DB設計（テーブル定義・ER図）はここに含めない。`docs/DB設計.md` として単独で出力する
> （作り方は `references/DB設計.md` を参照）。

---

## 対象と対象外（画面設計書との境界）

| ドキュメント | 記載すること |
|------------|------------|
| 画面設計書 | ある画面を操作したときに何が起きるか（表示仕様・操作・入力バリデーション・**どのAPIを使うかの一覧**） |
| 詳細設計（このドキュメント） | API・バッチ・RPA・CSV・外部連携・共通処理の**技術的な仕様そのもの**（リクエスト/レスポンス、実行トリガー、処理内容など）。画面から呼ばれるAPIであっても、技術仕様の実体はこちら側に書く |

> [!NOTE]
> CSVは輸出（ダウンロード）・輸入（アップロード）のどちらも詳細設計側に集約する。
> 画面設計書側は「CSVインポートボタンがある」「一覧をCSV出力できる」程度のUIレベルの記載に留め、
> 文字コード・必須カラム・エラー時の挙動などの技術仕様は詳細設計の該当CSVファイルに書く（相互リンクする）。

---

## 出力構成

出力先は `docs/詳細設計/` フォルダ。**カテゴリ内の個別項目ごとに1ファイル**にする。

```
docs/詳細設計/
├── index.md                      # カテゴリ別の一覧・相互リンク
├── API_ユーザー管理.md            # APIはリソース/コントローラー単位でまとめる
├── API_注文管理.md
├── Batch_日次売上集計.md          # バッチはジョブ単位
├── RPA_請求書自動配信.md          # RPAは処理単位
├── CSV_注文一覧.md                # CSVはファイル単位（出力・入力どちらも）
├── CSV_会員インポート.md
├── 外部連携_ヤマトAPI.md          # 外部連携は連携先単位
├── 外部連携_Slack.md
├── 外部連携_S3.md
├── 共通処理_認証.md               # 共通処理は関心事単位
├── 共通処理_監査ログ.md
└── 共通処理_例外処理.md
```

ファイル名は `<カテゴリ>_<項目名>.md`（画面設計書の `P01_画面名.md` と同じ考え方）。
**一括作成でよい**（画面設計書のように1項目ずつ確認しながら進める必要はない。項目数が多く、
往復すると非効率なため）。ただし作成後に一覧をユーザーに提示し、過不足があれば教えてもらう。

## 各カテゴリの作り方

### API（`API_<リソース名>.md`）

リソース・コントローラー単位でファイルを分け、その中に複数のエンドポイントをまとめる。

```markdown

[← 詳細設計一覧に戻る](./index.md)

**最終更新**: YYYY-MM-DD  
**バージョン**: v1.0

---

### [エンドポイント名]
- **メソッド**: GET / POST / PUT / DELETE
- **パス**: `/api/v1/xxx`
- **概要**: 
- **認証**: 必要 / 不要
- **利用画面**: [P01_xxx.md](../画面設計書/P01_xxx.md) のように使用元の画面があればリンクする（無ければ「バッチ/外部連携からのみ利用」等と明記）

#### リクエスト
| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|

#### レスポンス
```json
{
  "例": "レスポンスJSON"
}
```

#### エラーレスポンス
| ステータスコード | メッセージ | 説明 |
|----------------|-----------|------|

（同じリソース内の他のエンドポイントも同じフォーマットで続ける）
```

---

- **実行トリガー**: cron式 / 手動実行 / イベント起点 等
- **実行間隔・実行時刻**: 
- **処理概要**:

- **処理内容**: 
- **出力**: 更新されるテーブル・生成されるファイル等

- **実行環境**: （例：cronサーバー、Lambda、キューワーカー等）

### RPA（`RPA_<処理名>.md`）

```markdown
# RPA: [処理名]

[← 詳細設計一覧に戻る](./index.md)

**最終更新**: YYYY-MM-DD  
**バージョン**: v1.0

---

- **対象業務**: 
- **トリガー**: 定期実行 / 手動起動 / イベント起点
- **操作対象システム**: （例：〇〇の管理画面、Excelファイル）
- **処理フロー**: 手順を順番に記載する
- **異常時の挙動**: システムエラー・想定外画面が出た場合の扱い
```

### CSV（`CSV_<CSV名>.md`）

出力・入力の両方、または片方だけを記載する（存在する方だけでよい）。

```markdown
# CSV: [CSV名]（例：注文一覧.csv）

[← 詳細設計一覧に戻る](./index.md)

**最終更新**: YYYY-MM-DD  
**バージョン**: v1.0

---

- **方向**: 出力（ダウンロード） / 入力（インポート） / 両方
- **関連する画面**: [P01_xxx.md](../画面設計書/P01_xxx.md)（UIトリガーがあれば。無ければ「APIのみから利用」等）
- **文字コード**: UTF-8 / Shift-JIS
- **1行目（ヘッダー）**: あり / なし

## 出力仕様（方向が出力・両方の場合）

| 列番号 | ヘッダー名 | 内容 | 例 |
|--------|-----------|------|----|
| 1 | 注文ID | 注文を識別する番号 | 1001 |

**出力対象のデータ範囲**: （例：画面で絞り込んだ検索結果をそのまま出力 / 全件出力 等）

## 入力仕様（方向が入力・両方の場合）

| No | 項目名 | 必須 | 型・形式 | その他ルール |
|----|--------|------|---------|------------|

| 項目 | 内容 |
|------|------|
| 最大行数 | 〇〇件まで |
| 必須カラム | カラム名A, カラム名B |
| エラー時の挙動 | 全件ロールバック / エラー行スキップ 等 |
```

## 更新時の手順

詳細設計はファイルが項目ごとに分かれているため、更新は該当ファイルだけを触ればよい。

### 0. 旧形式（単一ファイルの `docs/詳細設計書.md`）からの移行

`docs/詳細設計/` フォルダが無く、代わりに旧い単一ファイル `docs/詳細設計書.md`
（API仕様・データベース設計・CSV出力仕様等が1ファイルにまとまった形式）が見つかった場合、
**「更新して」という指示であっても、まずこの移行を行ってから更新内容を反映する。**
旧形式のまま追記して済ませてはいけない。

1. `docs/詳細設計/` フォルダを作り、旧ファイルの「API仕様」セクションをリソース単位に分割して
   `API_<リソース名>.md` として保存する
2. 旧ファイルの「データベース設計」セクションは詳細設計に含めず、`docs/DB設計.md` として
   別ファイルに切り出す（`references/DB設計.md` の形式に従う）
3. 旧ファイルの「CSV出力仕様」セクションは `CSV_<CSV名>.md` として保存する。あわせて
   画面設計書側にCSVインポートの技術仕様が書かれていないか確認し、あれば `CSV_<CSV名>.md`
   に統合して画面設計書側はUIレベルの記載＋リンクに置き換える
4. コードを確認し、旧ファイルに無かった Batch・RPA・外部連携・共通処理があれば新規に追加する
5. `docs/詳細設計/index.md` を作成し、移行したファイルをカテゴリ別に一覧化する
6. 元の `docs/詳細設計書.md` を削除してよいかユーザーに確認してから削除する
7. 移行した旨（旧1ファイル → 新形式で何件のファイルに分割したか）をユーザーに報告する

### 通常の更新（すでに新形式になっている場合）

1. コードの変更箇所から、API・バッチ・RPA・CSV・外部連携・共通処理の追加・変更・削除を検出する
2. **新規追加された項目**: 新しいファイル（例：`API_在庫管理.md`）を作成する（最終更新・バージョンは `v1.0`）
3. **既存項目の内容変更**: 該当ファイルだけを上書き更新する（他のファイルには触れない）。
   その最終更新日を今日の日付に、バージョンを1つ上げる（v1.0 → v1.1）。スタンプが無ければ追加する
4. **削除された項目**: 該当ファイルを削除する（「廃止」として残さない。詳細設計は現在のシステムの姿を正確に表すことを優先する）
5. `docs/詳細設計/index.md` のカテゴリ一覧と、`index.md` 自身の最終更新日・バージョンを更新する。
   変更内容は `docs/変更履歴.md` に記録する（`references/変更履歴.md` 参照）

### 「レイアウトを最新にして」と言われた場合（コード変更が無い場合）

コード側の変更点が無く、レイアウトの整合性だけを確認・修正してほしいと言われた場合は、
上記の「新規項目の検出」は行わず、`docs/詳細設計/index.md` と**既存の全項目ファイル**
（`API_*.md` `Batch_*.md` `RPA_*.md` `CSV_*.md` `外部連携_*.md` `共通処理_*.md`）を
1つずつ次の観点でチェックする（件数が多くても全件チェックする。内容の書き換えは行わない）：

- `**最終更新**: YYYY-MM-DD` / `**バージョン**: vX.X` のスタンプがあるか。無ければ追加する
  （経緯が分からない場合は今日の日付・`v1.0` から追跡を開始する）
- ファイル自体に `## 変更履歴` のようなセクションが残っていないか。残っていれば
  `references/変更履歴.md` の移行手順に従い `docs/変更履歴.md` に移してから削除する

一致していたファイルには触れない。実際に直したファイルだけ最終更新日・バージョンを更新し、
`docs/変更履歴.md` に「詳細設計のレイアウトを最新化」のように短く記録する。

## この仕様書の記載漏れチェック観点

- 全APIエンドポイントがいずれかの `API_*.md` に記載されているか
- 定期実行されるバッチ・RPAがすべて `Batch_*.md` / `RPA_*.md` として記載されているか
- CSV入出力機能があるものはすべて `CSV_*.md` として記載され、画面設計書側からもリンクされているか
- 外部サービス・APIとの連携がすべて `外部連携_*.md` として記載されているか（認証情報はマスキングされているか）
- 認証・監査ログ・例外処理などの横断的な処理が `共通処理_*.md` として記載されているか
- `index.md` のカテゴリ一覧と実際に存在するファイルが一致しているか
- 旧形式の単一ファイル `docs/詳細設計書.md` が見つかった場合、新形式に移行してから削除確認を取っているか（旧形式のまま追記していないか）
