# コーディング規約の作り方

> [!IMPORTANT]
> このドキュメントは**ユーザーが明示的に要望したときだけ**作成する。他の仕様書のように
> デフォルトで作成したり、特定のファイルの存在から自動作成したりしない。

## 性格：あるべき論ではなく「実態の記録」

このドキュメントは「一般的なベストプラクティス」を書くものではない。**このプロジェクトのコードが
実際にどう書かれているか**をコードから読み取り、それを規約として明文化する。

- 同じ種類の処理（例：バリデーション）が複数箇所にある場合、**多数派のパターン**を規約として採用する
- 少数派のパターン（規約から外れているコード）は削除・修正の指示はせず、規約本文には書かない。
  該当箇所は `[要確認]` として、どこがどうブレているかを一覧化する（「規約違反リスト」のような
  別建ての仕組みは作らない。既存の `[要確認]` の運用に統一する）
- 判断材料が乏しい（実装例が1つしかない、パターンが割れて多数派が決められない）場合も `[要確認]` にする

---

## カバーする観点

以下の7観点で分析する。フレームワーク・言語は STEP 1 で読み込んだソースコードから判断し、
その構成に合わせて具体的な記述をする（Laravelなら「Request」「Service」「Repository」のような
実際のクラス名・ディレクトリ名を使う）。

1. **入力値検証**: バリデーションをどこで行っているか（専用クラス／ミドルウェア／コントローラ内 等）
2. **データ取得・永続化**: DBアクセスをどこで行っているか（Repository層／ORM直接呼び出し／QueryBuilder 等）
3. **ビジネスロジック・加工処理**: 独自ロジックやデータ加工をどこに書いているか（Service／UseCase／コントローラ内 等）
4. **コントローラの責務**: コントローラが何をして、何をしていないか
5. **命名規則**: クラス名・メソッド名・ファイル名・変数名の付け方
6. **ディレクトリ構成**: レイヤーごとのファイル配置場所
7. **エラーハンドリング**: 例外・エラーの扱い方、共通化の方法

---

## 分析の進め方

1. 各観点について、実装例を**複数箇所**（最低3〜5箇所）横断的に確認する。1箇所だけを見て決めつけない
2. パターンが揃っている場合は、そのパターンをコード例つきで規約として記載する
3. パターンが割れている場合は、件数の多い方を規約として採用しつつ、少数派の存在を `[要確認]` に記録する
4. 該当する実装が見つからない観点（例：エラーハンドリングの共通化が特に存在しない）は「特に統一された方針は見られない」と正直に書き、`[要確認]` にする
