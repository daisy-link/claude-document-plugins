# システム仕様書作成スキル

対象プロジェクトのソースコードを読み込み・分析して、誰が見てもわかるMarkdown形式のシステム仕様書を自動で作成する。

このSKILL.mdは**全体の進行（モード判定・作成順序・index生成・記載漏れチェック・共通ルール）**を扱う。
各仕様書の詳しい作り方とテンプレートは `references/` に分かれており、**その仕様書を作るときだけ該当ファイルを読み込む**。

> [!NOTE]
> このスキルは Claude Code 上で動く。分析対象は**現在の作業ディレクトリ（リポジトリ）**、
> 出力先はそのプロジェクト内の `docs/` フォルダ。ユーザーが別のパスを指定した場合はそれに従う。

---

## ⚠️ 絶対遵守：記載禁止事項

**以下の情報はいかなる仕様書・ファイルにも絶対に記載しない。** インプットのファイルに含まれていても出力には含めない。

### 記載禁止の情報

| 種別 | 具体例 |
|------|--------|
| 認証情報 | パスワード、APIキー、アクセストークン、シークレットキー、秘密鍵 |
| 接続情報 | DB接続文字列（ユーザー名・パスワード含むもの）、内部IPアドレス、内部ホスト名 |
| 個人情報 | 氏名、メールアドレス、電話番号、住所、生年月日、マイナンバー等の個人を特定できる情報 |
| その他機密情報 | 本番環境の具体的なURL・エンドポイント、ライセンスキー、証明書の内容 |

### 代替表記ルール

禁止情報が必要な箇所は以下の形式でマスキングして記載する：

| 種別 | 記載方法 |
|------|---------|
| パスワード | `********` または `[設定値 - 記載不可]` |
| APIキー・トークン | `[APIキー - 記載不可]` |
| 個人情報 | `[個人情報 - 記載不可]` |
| DB接続文字列 | `[接続先DB名]://[ユーザー名]:[パスワード]@[ホスト]/[DB名]` のように構造のみ示す |
| 具体的な値が必要な場合 | `.env.example` や設定ファイルのサンプルを参照するよう案内する |

> 💡 **値ではなく「何の設定か・変更するとどうなるか」の説明に徹すること。**

## 対応する仕様書の種類

各仕様書の詳しい作り方・テンプレート・記載漏れチェック観点は、右列の参照ファイルにある。
**その仕様書を作成する段階で該当ファイルを読み込み、その指示に従う。**

| 種類 | 内容 | 参照ファイル |
|------|------|------------|
| 要件定義書 | 「このシステムは何のために存在するか」「何ができるか」。機能一覧・優先度・制約事項 | `references/要件定義書.md` |
| システム構成書 | システム全体の構成図・環境構成・サービス構成・DB接続 | `references/システム構成書.md` |
| 画面設計書 | 画面一覧・画面フロー・画面ごとの詳細仕様・データフロー（**画面がある場合のみ作成**。単一ファイルではなく `docs/画面設計書/` フォルダ構成） | `references/画面設計書.md` |
| 詳細設計書 | API仕様・DBテーブル設計・CSV出力項目 | `references/詳細設計書.md` |
| システム設定仕様書 | アプリ動作を制御する設定値の一覧と、変えると何が変わるか | `references/システム設定仕様書.md` |
| インフラ構成書 | サーバー・クラウド・ネットワーク構成・デプロイ手順 | `references/インフラ構成書.md` |
| 開発環境セットアップ | 手元PCで動かすための環境構築手順（Docker等） | `references/開発環境セットアップ.md` |
| コーディング規約 | 実際のコードから読み取れる実装パターン（バリデーション・DB接続・ロジックの置き場所等）の明文化。**ユーザーが要望したときだけ作成する（オプトイン）** | `references/コーディング規約.md` |

> [!NOTE]
> 旧「基本設計書」は**システム構成書**（システム構成図・環境構成）と**画面設計書**（画面まわり）に分割されている。
> 「基本設計書を作って」と言われた場合はこの2つをまとめて指すものとして扱う。

## 動作モードの判定（最初に確認）

ユーザーのリクエストが **新規作成** / **更新** / **再作成** かを最初に判定する。

| 判定条件 | モード |
|---------|--------|
| `docs/` フォルダや既存の仕様書ファイルが存在しない | **新規作成モード** → STEP 1へ |
| 「仕様書を更新して」「コードを修正した」「追記して」などの発言がある | **更新モード** → 更新フローへ |
| 既存の仕様書ファイルをアップロードしている | **更新モード** → 更新フローへ |
| 「最初から作り直して」「全部削除して作り直して」「リセットして」などの発言がある | **再作成モード** → 再作成フローへ |

---

## 【再作成モード】

### R-STEP 1: 既存ファイルの削除確認

既存の `docs/` フォルダを削除する前に、必ずユーザーに確認する：

> 「`docs/` 内のファイルをすべて削除して最初から作り直します。よろしいですか？」

確認が取れたら削除を実行する：

```bash
rm -rf docs/
mkdir -p docs/
```

### R-STEP 2: 新規作成モードで再実行

削除完了後は **新規作成モードと同じ手順（STEP 1〜4）** で仕様書を最初から生成する。

---

## 【更新モード】

### U-STEP 1: 変更内容の把握

以下の2つの方法で変更点を把握する。**両方指定された場合は両方を使う。**

**方法A：修正後のプロジェクトファイルから差分を検出する**

```bash
# 変更のあったファイルを把握（git管理下なら差分から、そうでなければ全体を確認）
git status --short 2>/dev/null || find . -type f -not -path './.git/*' -not -path './docs/*' | sort
```

修正後ファイルと既存仕様書（`docs/`）を読み込み、**何が変わったか**を自律的に検出する。検出観点：

- APIエンドポイントの追加・変更・削除
- テーブル定義・カラムの追加・変更・削除
- 環境変数・設定値の変更
- 依存パッケージのバージョン変更
- Dockerサービスの追加・変更

**方法B：ユーザーの口頭説明から変更点を把握する**

ユーザーが「〇〇を追加した」「△△のAPIを変更した」と伝えてきた場合、その説明をそのまま変更点として受け取り、該当箇所を仕様書に反映する。

### U-STEP 2: 変更箇所の特定と更新

1. 変更内容に関係する仕様書ファイルを特定する
2. 既存の仕様書を読み込む（`docs/` 以下）
3. 変更箇所のみを上書き更新する（関係のない箇所は変更しない）
4. `index.md` の **最終更新日** と **更新履歴** を更新する

> [!IMPORTANT]
> **画面設計書だけは通常の上書き更新と手順が異なる。** `references/画面設計書.md` の更新時の
> 手順（新規画面のみ1画面ずつ確認しながら追加、既存画面は確認なしで自動同期、削除された画面は
> 記載ごと削除）に従うこと。

### U-STEP 3: 変更履歴の記録

各仕様書ファイルの先頭に変更履歴セクションを追記・更新する：

```markdown
## 変更履歴

| 日付 | バージョン | 変更概要 |
|------|----------|---------|
| YYYY-MM-DD | v1.1 | [変更内容の要約] |
| YYYY-MM-DD | v1.0 | 初版作成 |
```

`index.md` にも変更サマリーを追記する：

```markdown
## 更新履歴

| 日付 | 更新ドキュメント | 変更概要 |
|------|--------------|---------|
| YYYY-MM-DD | 詳細設計書 | 〇〇APIを追加、△△テーブルにカラム追加 |
| YYYY-MM-DD | 全ドキュメント | 初版作成 |
```

### U-STEP 4: 更新結果の報告

```
【更新完了】
- 更新したファイル: 詳細設計書.md
- 変更箇所:
  - API仕様 > POST /api/items を追加
  - DB設計 > items テーブルに stock カラムを追加
- index.md の更新履歴に記録しました
```

## STEP 1: 対象プロジェクトのファイル確認

分析対象は**現在の作業ディレクトリ（リポジトリ）**。まずどんなファイルがあるかを確認してから内容を読み込む。
ユーザーが特定のサブディレクトリを指定した場合はそこを対象にする。

```bash
# ディレクトリ構造の全体把握（.git や出力先の docs は除外）
find . -type f -not -path './.git/*' -not -path './docs/*' | sort
```

### ファイルの種類ごとの読み方

| ファイル種別 | 読み方 |
|------------|--------|
| `.py` `.js` `.ts` `.java` `.go` など | `cat`（20KB以下）or `head -100` + `grep`で要所抽出 |
| `requirements.txt` `package.json` `pom.xml` | `cat`（依存関係・バージョンを把握） |
| `.yaml` `.yml` `.json` `.env` `.toml` | `cat`（設定値・環境変数を把握） |
| `.sql` `schema.*` `models.*` | `cat`（テーブル定義・リレーションを把握） |
| `README*` `*.md` | `cat`（既存ドキュメントを活用） |
| `Dockerfile` `docker-compose.*` | `cat`（インフラ構成を把握） |
| `.docx` `.pptx` `.pdf` `.xlsx`（企画書・仕様メモ・ER図等） | 内容を確認して読み取る（開発前モードの主要な入力） |
| 画像ファイル（ER図・ワイヤーフレームのスクリーンショット等） | 画像として直接内容を確認する |

### 開発前モード（実装コードがまだ無い場合）

このスキルは既存コードの分析だけでなく、**システム開発着手前の設計書作成**にも使う。
以下のいずれかに該当する場合は「開発前モード」として扱う。

- ユーザーが「開発前です」「まだ実装していません」「これから作るシステムです」「要件定義から」等と発言している
- ディレクトリを確認した結果、アプリケーションのソースコード（エントリーポイント・ルーティング・モデル等）がほとんど無く、
  代わりに企画書・ヒアリングメモ・ER図・ワイヤーフレームなど**設計段階の資料**が見つかる

開発前モードでは、各参照ファイル（`references/*.md`）の指示にある**「コードから読み取る」は
「企画書・設計資料から読み取り、それも無ければユーザーに質問する」に読み替える**。
対話中の「コードからは〇〇と読み取れました」という前置きは、情報源が無ければ省略し、直接質問してよい。

> [!IMPORTANT]
> **コーディング規約だけは開発前モードで作成できない**（実装コードから実際のパターンを抽出するドキュメントのため）。
> ユーザーがこの段階でコーディング規約を要望した場合は、「実装コードがまだ無いため、規約は最初の実装がある程度進んでから作成することをおすすめします」と案内する。

開発前モードでも他の6種類（要件定義書・システム構成書・画面設計書・詳細設計書・システム設定仕様書・
インフラ構成書・開発環境セットアップ）はすべて作成対象になる。この場合は「予定している構成・仕様」として記載し、
実装段階で内容が変わり得ることが分かるようにする（例：見出しに「（予定）」と付ける、または冒頭に一言添える）。

### 追加ドキュメントを自動作成する条件

以下を検出したら、対応する仕様書を**自動的に作成対象に加える**。

```bash
# 設定ファイルの存在チェック
find . -not -path './.git/*' \( -name ".env*" -o -name "*.env" \
  -o -name "config.*" -o -name "settings.*" \
  -o -name "application.yml" -o -name "application.properties" \
  -o -name "appsettings.json" \) | sort

# Docker等の開発環境ファイルの存在チェック
find . -not -path './.git/*' \( -name "Dockerfile" -o -name "docker-compose*.yml" \
  -o -name "docker-compose*.yaml" -o -name ".devcontainer" \
  -o -name "Vagrantfile" -o -name "Makefile" \) | sort
```

| 検出したもの | 自動作成する仕様書 |
|------------|-----------------|
| `.env` `config.*` `settings.*` `application.yml` 等、または DBに `m_config` `settings` `system_config` 等の設定系テーブル | **システム設定仕様書** |
| `Dockerfile` `docker-compose.*` `.devcontainer/` `Vagrantfile` `Makefile` `.env.example` 等 | **開発環境セットアップ** |

### 分析時の注意点
- まずディレクトリ構造を見てプロジェクト全体像をつかむ
- 大量のファイルがある場合は重要ファイルを優先（エントリーポイント・設定・DB定義）
- コードから**実装の意図**を読み取って仕様に落とし込む

## STEP 2: 作成する仕様書の種類を確認

ユーザーから指定がある場合はそれに従う。指定がない場合はアップロードされたファイルの内容から判断して提案する。

複数種類を作る場合は、以下の順序で作成する：

1. 要件定義書（作成前に**対話形式 / 自動生成**をユーザーに選んでもらう。詳細は `references/要件定義書.md`）
2. システム構成書
3. 画面設計書（**画面がある場合のみ**。無い場合はスキップする）
4. 詳細設計書
5. システム設定仕様書（設定ファイルまたは設定系マスターテーブルがある場合は**自動作成**）
6. インフラ構成書
7. 開発環境セットアップ（Docker等の開発環境ファイルがある場合は**自動作成**）
8. コーディング規約（**ユーザーが明示的に要望した場合のみ**。デフォルトの作成対象には含めない・自動検出もしない。**要望があれば必ず最後に配置**。**開発前モードでは作成できない**）

> [!NOTE]
> 画面設計書は他の仕様書と作り方が異なる（一覧を確定してから1画面ずつ作る）。詳細は `references/画面設計書.md` を参照。

## STEP 3: 仕様書の作成

作成する各仕様書について、**「対応する仕様書の種類」の表に示した参照ファイルを読み込み、その指示とテンプレートに従って生成する。**
共通の書式ルールは以下のとおり（全仕様書に適用する）。

> [!IMPORTANT]
> **画面設計書だけは一括生成しない。** `references/画面設計書.md` に記載された専用の手順
> （画面一覧を確定 → 1画面ずつ作成して都度確認）に従うこと。それ以外の仕様書は通常どおり
> 参照ファイルのテンプレートに沿って一括で作成してよい。

### 強調表現のルール（全仕様書共通）

重要な箇所・注目してほしい箇所は以下で統一して強調する。**絵文字は使わない。**

#### 太字（`**テキスト**`）
文章中の重要な語句・値・項目名を強調するときに使う。

```markdown
- **必須**: 全ユーザーが登録時に入力しなければならない
- このAPIは**認証が必須**です
- **最大100件**まで一括登録できる
```

#### GitHub Alerts記法
ブロック単位で注意・補足・重要事項を目立たせるときに使う。

| 記法 | 用途 |
|------|------|
| `> [!NOTE]` | 補足説明・読んでおくと役立つ情報 |
| `> [!TIP]` | 推奨する使い方・便利な情報 |
| `> [!IMPORTANT]` | 必ず読んでほしい重要な情報 |
| `> [!WARNING]` | 設定ミス・操作ミスに注意が必要な情報 |
| `> [!CAUTION]` | 取り返しのつかない操作・データ削除などの危険な情報 |

```markdown
> [!IMPORTANT]
> このAPIを呼び出す前に必ず認証トークンを取得してください。

> [!WARNING]
> この設定を変更すると、全ユーザーのセッションが即時無効になります。

> [!CAUTION]
> 削除操作は取り消しできません。実行前にバックアップを取得してください。

> [!NOTE]
> 開発環境では`DEBUG=true`にすると詳細なエラーが確認できます。
```

#### 使い分けの基準

| 状況 | 使う表現 |
|------|---------|
| 文章中の重要な語句を目立たせたい | **太字** |
| 「絶対に忘れないでほしい」重要事項 | `[!IMPORTANT]` |
| 「間違えると困る」注意事項 | `[!WARNING]` |
| 「データ消失・障害につながる」危険な操作 | `[!CAUTION]` |
| 「知っておくと便利」な補足情報 | `[!NOTE]` |
| 「こうするとより良い」推奨事項 | `[!TIP]` |

## STEP 4: ファイルの保存と提示

### 出力先の決定（必須）

出力先はプロジェクト直下の `docs/`。作業ディレクトリ内に作成する。

```bash
# プロジェクト直下に docs/ を用意する
mkdir -p docs
```

> 生成物はリポジトリ内の `docs/` に直接書き込まれる。`.git` と同じ階層に `docs/` ができる。
> ユーザーが別の出力先を指定した場合はそこに従う。

### 出力ルール

- ファイル名は **資料名のみ** とする。システム名・プロジェクト名は絶対に含めない。

| 正しいファイル名 | 間違いの例（禁止） |
|---------------|----------------|
| `要件定義書.md` | `要件定義書_システム名.md` |
| `システム構成書.md` | `システム構成書_プロジェクト名.md` |
| `詳細設計書.md` | `詳細設計書_ふるさと納税返礼品マスタ整合性チェックツール.md` |
| `コーディング規約.md` | `コーディング規約_プロジェクト名.md` |

> [!CAUTION]
> ファイル名にアンダースコア（`_`）やシステム名を付けてはいけない。資料名＋`.md` のみ。
> **例外は画面設計書のみ**：`docs/画面設計書/` フォルダの中に `P01_ログイン画面.md` のように
> 「No_画面名.md」形式で保存する（これはシステム名ではなく画面の識別のためのアンダースコアなので許可する）。

- 複数の仕様書を作る場合はすべて `docs/` 以下に出力し、生成後にファイルパスをまとめてユーザーに提示する（クリックで開ける相対パスで示す）
- コードから読み取れなかった項目は `[要確認]` と明記する

## STEP 5: 記載漏れチェック（必須）

すべての仕様書を作成・保存したあと、必ず実行する。
**プログラムのファイルと作成した仕様書を見比べて、書き忘れがないか確認し、漏れがあれば即座に追記する。**

### C-STEP 1: 作成した仕様書とプログラムファイルを再度読み込む

```bash
find docs/ -name "*.md" | sort
find . -type f -not -path './.git/*' -not -path './docs/*' | sort
```

### C-STEP 2: 仕様書ごとに「書き漏れがないか」を確認する

各仕様書の**具体的なチェック観点は、それぞれの参照ファイル（`references/[仕様書名].md`）末尾の「記載漏れチェック観点」に記載してある。**
作成した仕様書ごとに、対応する参照ファイルのチェック観点に照らしてプログラムと照合し、漏れを確認する。

### C-STEP 3: 書き漏れが見つかったら追記する

漏れが見つかった場合は、該当する仕様書ファイルを直接上書きして追記する。追記内容が多い場合はユーザーに「〇〇について追記しました」と報告する。

### C-STEP 4: チェック結果をユーザーに伝える

```
【記載漏れチェック完了】

✅ 漏れなし:
  - 要件定義書
  - インフラ構成書

📝 追記あり:
  - 詳細設計書
    → GET /api/items/{id} エンドポイントを追記
    → items テーブルの updated_at カラムを追記
  - 画面設計書
    → 設定画面（/admin/settings）が未記載だったため追記
```

漏れが一切なかった場合は「全仕様書の記載漏れチェック完了。追記箇所はありませんでした。」と報告する。

## 品質チェックリスト

仕様書を作成したあと、以下をすべて確認する（各仕様書固有の観点は参照ファイル末尾を参照）：

- [ ] 重要な箇所に **太字** または GitHub Alerts記法が使われている
- [ ] ⚠️ パスワード・APIキー・トークン等の認証情報が一切記載されていない
- [ ] ⚠️ 個人情報（氏名・メール・電話番号・住所等）が一切記載されていない
- [ ] ⚠️ 本番DBの接続文字列・内部IPアドレス等の機密接続情報が記載されていない
- [ ] システム名・目的が明記されている
- [ ] 各機能が具体的に説明されている（あいまいな記述がない）
- [ ] `[要確認]` 箇所をユーザーに伝えている
- [ ] Mermaid記法が正しい（構文エラーなし）
- [ ] 要件定義書を作成する前に対話形式・自動生成のモード選択をユーザーに確認している
- [ ] 画面がある場合、画面設計書は一覧を確定してから1画面ずつ確認しながら作成している（一括生成していない）
- [ ] コーディング規約は、ユーザーが明示的に要望していないのに作成していない（オプトイン）
- [ ] **（開発前モード）** コーディング規約を作成しようとしていない（作れない旨を案内している）
- [ ] **（開発前モード）** 情報源が企画書・設計資料・対話であることが分かる（存在しないコードを読んだかのような記述になっていない）
- [ ] 設定ファイルまたは設定系マスターテーブルがある場合、システム設定仕様書が生成されている
- [ ] Docker等の開発環境ファイルがある場合、開発環境セットアップが生成されている
- [ ] STEP 5の記載漏れチェックを実施し、結果をユーザーに報告している
- [ ] `docs/index.md` が生成され、作成した全ファイルへのリンクが含まれている
- [ ] `index.md` の `[要確認]` まとめに各ドキュメントの要確認事項が集約されている
- [ ] **（更新モード）** 変更箇所のみ上書きし、無関係な箇所を書き換えていない
- [ ] **（更新モード）** 各仕様書に変更履歴セクションが追記されている
- [ ] **（更新モード）** `index.md` の更新履歴・最終更新日が更新されている
- [ ] **（再作成モード）** 削除実行前にユーザーへの確認を取っている
