# 要件定義書の作り方

「このシステムは何のために存在するか」「何ができるか」をまとめた文書。
機能の一覧・優先度・制約事項を記載する。

> [!IMPORTANT]
> 要件定義書は他の仕様書と異なり、**コードだけでは読み取れない「なぜ作ったか・誰のためか」という背景や意図**が重要になる。
> そのため、まず作成モード（対話形式 / 自動生成）をユーザーに選んでもらってから進める。

---

## 作成モードの選択（必ず最初に行う）

> [!IMPORTANT]
> 要件定義書の作成を始める前に、**必ず最初にモードを選んでもらう**。
> ユーザーがすでに「仕様書を作って」とだけ言っている場合でも、要件定義書に入る前に必ず以下の確認メッセージを出すこと。
> モード選択をスキップして自動で進めてはいけない。

ユーザーへの確認メッセージ（そのまま使う）：

```
要件定義書の作成方法を選んでください。

  1. 対話形式（推奨）
     コードを分析した内容をベースに、質問に答えながら一緒に作成します。
     背景・目的・優先度など、コードだけでは読み取れない情報も正確に反映できます。

  2. 自動生成
     コードを分析して自動で作成します。
     読み取れなかった項目は [要確認] として記載します。

「1」または「2」でお答えください。
```

- ユーザーが **「1」「対話形式」「対話」** などと答えた場合 → 「対話形式フロー」へ
- ユーザーが **「2」「自動」「自動生成」** などと答えた場合 → 「自動生成フロー」へ
- ユーザーが返答せずに別の話をしてきた場合 → もう一度モード選択を促す

---

## 【対話形式フロー】

コードを分析した内容をベースにしながら、ユーザーと確認・補足を重ねながら作成する。

**Q-1 〜 Q-4 の順に1つずつ質問する。** 前の回答を受けてから次の質問に進む。まとめて聞かない。

---

**Q-1：システム概要・業務背景・外部システムの確認**

まずコードを分析して読み取れた内容を以下の形式で提示し、合っているか・補足があるかを確認する：

```
コードを分析しました。以下の内容で合っていますか？

- システム名: [コードから推測した名前]
- このシステムの目的: [コードから読み取った目的]
- 主に使うのはどんな人: [コードから推測したユーザー像]
- 連携している外部システム: [コードから読み取れたもの / なし / 不明]

補足や修正があれば教えてください。
また、以下についても教えていただけると仕様書に反映できます：
・このシステムを作った業務上の背景・きっかけ（例：手作業の効率化、既存システムのリプレイス など）
・このシステムが解決したい業務上の課題
・上記以外に連携している外部システム・サービスがあれば教えてください
```

---

**Q-2：機能一覧の確認**

コードから読み取れた機能をリストアップして提示し、漏れや優先度を確認する：

```
コードから以下の機能が確認できました。

| No | 機能名 | 概要 | 優先度（仮） |
|----|--------|------|------------|
| F01 | [機能名] | [概要] | 高/中/低 |

・優先度の認識は合っていますか？
・リストに載っていない機能はありますか？
・現時点では未実装だが、将来追加予定の機能はありますか？
```

---

**Q-3：非機能要件の確認**

コードや設定ファイルから読み取れた内容を提示し、目標値を確認する：

```
非機能要件について確認させてください。

コードから以下の情報が確認できました：
- 認証方式: [読み取れた内容 / 不明]
- レート制限: [読み取れた内容 / 不明]

以下の点について、決まっている目標値や要件があれば教えてください：
・レスポンスタイムの目安（例：3秒以内）
・同時アクセス数の想定（例：最大100人）
・稼働率の目標（例：99.9%）
・セキュリティで特別な要件（例：2段階認証必須）
```

---

**Q-4：制約事項・前提条件の確認**

```
最後に、制約事項や前提条件を確認させてください。

・使用できない技術や外部サービスの制限はありますか？
・対応が必要なブラウザ・デバイスの指定はありますか？
・法律・規制への対応（個人情報保護法・GDPR等）は必要ですか？
・既存システムとの連携や移行の制約はありますか？
```

全質問の回答が揃ったら、下記テンプレートに対話で得た情報を埋めてファイルを出力する。

---

## 【自動生成フロー】

コードを分析して読み取れる情報をすべて埋める。読み取れなかった項目は `[要確認]` と記載する。

> [!NOTE]
> 自動生成では背景・目的・優先度などがコードから読み取れず `[要確認]` になることが多い。
> 後から手動で補足するか、生成後に対話形式で修正することを推奨する。

## この仕様書の記載漏れチェック観点

- Q-1〜Q-4の全質問への回答が仕様書に反映されているか
- ユーザーから得た背景・補足情報が記載されているか
- コードに実装された機能が機能要件に全て挙がっているか
- 業務背景（背景・課題・期待する効果）が記載されているか
- 外部システムとの連携がある場合、外部システム一覧が記載されているか
