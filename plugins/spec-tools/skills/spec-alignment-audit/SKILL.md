# 仕様書・実装 整合性監査スキル

`docs/` の仕様書と実装済みのコードをプロジェクト全体で突き合わせ、両方向の乖離
（仕様書にあるがコードにない／コードにあるが仕様書にない）を洗い出し、**見つかったその場で
1件ずつユーザーに確認しながら解決していく。**

> [!IMPORTANT]
> このスキルは**独立して直接呼び出す**。`spec-dev-cycle` の内部スキルではない。
> `spec-review` は「直近実装した1機能」を対象にした開発サイクル内のレビューであり、
> 本スキルは**プロジェクト全体**を対象にした定期的な監査という違いがある。

> [!CAUTION]
> `../共通ルール.md` に spec-tools プラグイン全体の絶対禁止事項がある。解決のために
> `spec-implementation` や `architecture-design` を呼び出す場合も、それぞれの繰り返し上限・
> 禁止事項に従う。

---

## STEP 1: 前提確認（必須）

```bash
find docs/ -name "*.md" 2>/dev/null | sort
find . -type f -not -path './.git/*' -not -path './docs/*' -not -path './node_modules/*' | head -50
```

- `docs/` に仕様書が見つからない場合 → 実施せず、次のように案内して停止する：

  > 「`docs/` に仕様書が見つかりません。整合性監査には仕様書が必要です。
  > 先に architecture-design スキルで仕様書を作成しますか？」

- 実装コードが実質的に存在しない（開発前モードの設計書だけの状態）場合 → 実施せず、
  次のように案内して停止する：

  > 「まだ実装コードが無いようです。このスキルは仕様書と実装コードの両方が揃っている
  > プロジェクト向けです。実装が進んでから改めて実行してください。」

---

## STEP 2: 監査範囲の特定

- ユーザーが対象範囲（特定の画面・機能・モジュール）を指定した場合はそれに従う。
- 指定が無い場合は**プロジェクト全体**を対象にする（`docs/` 配下の全仕様書 × 実装コード全体）。
  範囲が広い場合は、`docs/画面設計書/`・`docs/詳細設計/` のカテゴリ単位で1つずつ順に処理してよい
  （一度に全部を読み込もうとしてトークンを浪費しない。`../共通ルール.md` のトークン節約方針）。

---

## STEP 3: 突き合わせ

仕様書の種類ごとに、対応する実装の有無を確認する。

| 仕様書 | 突き合わせる実装 |
|--------|----------------|
| 画面設計書（各画面） | 対応するルーティング・ページコンポーネントが実装されているか |
| 詳細設計（API） | 仕様書に書かれたエンドポイントが実装され、逆に実装済みのエンドポイントが全て仕様書にあるか |
| 詳細設計（バッチ・RPA） | 仕様書に書かれたジョブが実装され、逆に実装済みのジョブ定義が全て仕様書にあるか |
| 詳細設計（外部連携） | 仕様書に書かれた連携先が実装され、逆に実装コード内の外部API呼び出しが全て仕様書にあるか |
| DB設計 | 仕様書に書かれたテーブル・カラムが実際のスキーマにあり、逆に実際のスキーマの
  テーブル・カラムが全て仕様書にあるか |
| 要件定義書 | 機能一覧の各機能に対応する実装があるか |

**判断基準は「機能単位で明確に有無が言えるもの」に絞る。** 文言の細かい表現差・軽微な実装の
揺れは対象にしない（`spec-review` の規約チェックの役割であり、本スキルは存在の有無を見る）。

両方向をそれぞれリストアップする：

- **A. 仕様書にあるが実装されていない**（未実装）
- **B. 実装にあるが仕様書に記載が無い**（未記載）

---

## STEP 4: 1件ずつ確認して解決する

見つかった乖離を、A・Bそれぞれ順番に1件ずつ提示し、その場で対応を選んでもらう。
**まとめて一覧だけ見せて終わりにしない。**

### A. 未実装が見つかった場合

```
【未実装】画面設計書 P05_注文詳細画面.md に記載の「注文キャンセルボタン」が実装されていません。

どうしますか？
(1) 実装する（spec-implementation を呼び出します）
(2) 仕様から削除する（architecture-design で該当記載を削除します）
(3) 保留する（今回は何もしません）
```

- **(1) 実装する**：`../spec-implementation/SKILL.md` の手順でその場で実装する
  （実装後のレビューも内蔵されている）
- **(2) 仕様から削除する**：`../architecture-design/SKILL.md`（更新モード）でその仕様記載を削除する
- **(3) 保留する**：何もせず次の項目に進む。最終報告に「保留」として記録する

### B. 未記載が見つかった場合

```
【未記載】src/routes/export.ts に `GET /api/export/csv` が実装されていますが、
詳細設計に記載がありません。

どうしますか？
(1) 仕様書に追記する（architecture-design で該当仕様書に追記します）
(2) 保留する（今回は何もしません）
```

- **(1) 仕様書に追記する**：`../architecture-design/SKILL.md`（更新モード）でコードの内容から
  該当仕様書に追記する
- **(2) 保留する**：何もせず次の項目に進む。最終報告に「保留」として記録する

> [!NOTE]
> **コードを削除する選択肢は用意しない。** コードの削除はレビュー・リファクタリングの領域であり、
> 本スキルの役割外。未記載のコードが不要だと判断した場合は、ユーザー自身が別途対応する。

---

## STEP 5: 完了報告

すべての項目の確認が終わったら、対応状況をまとめて報告する。

```
【整合性監査 完了】
- 未実装として見つかった項目: 3件
  - 注文キャンセル機能（画面設計書 P05）→ 実装完了
  - 会員退会機能（要件定義書 F12）→ 仕様から削除
  - 一括CSVインポート（詳細設計 CSV_会員インポート）→ 保留
- 未記載として見つかった項目: 2件
  - GET /api/export/csv → 詳細設計に追記
  - 管理画面の一括承認機能 → 保留
```

## 品質チェックリスト

- [ ] `docs/` に仕様書が無い場合、実施せず architecture-design を案内している
- [ ] 実装コードが実質的に存在しない（開発前）場合、実施せず利用不可の旨を案内している
- [ ] 「仕様書にあるがコードにない」「コードにあるが仕様書にない」の両方向をチェックしている
- [ ] 軽微な表現差ではなく、機能単位で明確な有無に絞って乖離としている
- [ ] 見つかった乖離を一覧で提示するだけで終わらせず、1件ずつユーザーに確認して解決している
- [ ] 未実装の解決策（実装する/仕様から削除する/保留する）をその都度提示している
- [ ] 未記載の解決策（仕様書に追記する/保留する）をその都度提示している
- [ ] コードを削除する選択肢を提示していない（このスキルの役割外）
- [ ] 実装する場合は spec-implementation、仕様書を書き換える場合は architecture-design の
      手順に従っている（本スキル自身が直接 `docs/` やコードを書き換えていない）
- [ ] 対象に直接関連する仕様書・コードだけを読み込んでいる（`docs/` を無差別に全件読み込んでいない）
- [ ] 最終報告に、各項目の対応結果（実装/削除/追記/保留）を明記している
