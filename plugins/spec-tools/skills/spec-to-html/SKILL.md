# 仕様書 HTML 変換スキル

Markdown の仕様書を、**ネット接続なしでブラウザで開ける HTML** に変換する。
`system-spec-writer` が出力する `docs/` フォルダ（`index.md` + 各仕様書）をそのまま入力にできる。

変換は同梱の Python スクリプト `scripts/md_to_html.py` が行う。
このスクリプトは **外部ライブラリに依存せず**（Python標準ライブラリのみ）、
**Mermaid.js を同梱**しているため、実行環境やネット環境に左右されず確実に動く。

---

## このスキルが確実に処理すること

Markdown をそのまま HTML にするだけでなく、仕様書特有の以下の要素を正しく変換する。
ここが手作業や単純な変換ツールでは崩れやすいポイントなので、必ずこのスクリプトを使う。

| 要素 | 変換結果 |
|------|---------|
| Mermaid図（`graph` `stateDiagram` `sequenceDiagram` `erDiagram` 等） | ブラウザ上で図として描画（オフライン対応） |
| GitHub Alerts（`> [!NOTE]` `[!TIP]` `[!IMPORTANT]` `[!WARNING]` `[!CAUTION]`） | 色分けされたコールアウト |
| `.md` 同士の内部リンク（例 `[戻る](./index.md)`） | `.html` への相対リンクに自動で張り替え |
| GFMテーブル | 罫線・ゼブラ模様付きのHTMLテーブル |
| チェックリスト（`- [x]` / `- [ ]`） | ☑/☐ 付きの項目 |
| 見出し | ページ内目次（右側）とサイドバーに反映 |

出力は **仕様書ごとに1つの HTML**。`index.html` がポータルになり、
全ページ共通のサイドバーからどのドキュメントにも移動できる。

---

## 実行手順

### STEP 1: 入力の Markdown を特定する

変換元の `.md` がどこにあるかを確認する。多くはプロジェクト内の `docs/` フォルダ（`system-spec-writer` の出力先）。

```bash
# プロジェクト内の仕様書フォルダを探す
find . -type d -name docs 2>/dev/null

# フォルダ内の .md を確認
ls -1 docs/*.md 2>/dev/null
```

- `docs/` フォルダ（`index.md` を含む複数ファイル）が見つかれば、そのフォルダごと入力にする。
- ユーザーが別の場所やファイルを指定した場合はそれに従う。
- どこを変換するか曖昧なときはユーザーに確認する。

> [!NOTE]
> `index.md` があるとポータルページになり、サイドバーの先頭に並ぶ。
> `index.md` が無くても変換は動く（最初のファイルがトップになる）。

### STEP 2: 変換を実行する

同梱スクリプトを実行する。入力ディレクトリと出力ディレクトリを渡すだけ。
スクリプトはプラグインに同梱されているので、`${CLAUDE_PLUGIN_ROOT}` 経由で参照する（この環境変数は実行時に自動で展開される）。

```bash
python3 "${CLAUDE_PLUGIN_ROOT}/skills/spec-to-html/scripts/md_to_html.py" <入力ディレクトリ> <出力ディレクトリ>
```

出力先はプロジェクト内に `docs-site/`（または `html/`）を作るのが分かりやすい。

```bash
# 例: プロジェクト内の docs → docs-site
python3 "${CLAUDE_PLUGIN_ROOT}/skills/spec-to-html/scripts/md_to_html.py" docs docs-site
```

**オプション**

| オプション | 用途 |
|-----------|------|
| `--title "サイト名"` | サイドバー見出しのタイトルを指定（省略時は最初のH1から自動取得） |
| `--standalone` | Mermaid を各HTMLに埋め込み、**1ファイル単体**で完結させる。`assets/` フォルダ無しでどこでも開けるが、ファイルサイズは大きくなる |

> [!TIP]
> 既定では出力フォルダに `assets/mermaid.min.js` を1つ置き、各HTMLがそれを参照する。
> **`html/` フォルダごと**配布・オープンする使い方が最も軽量。
> HTMLファイルを1枚だけ単独で渡したい場合は `--standalone` を使う。

### STEP 3: 出力を確認して提示する

```bash
# 生成物を確認
find <出力ディレクトリ> -type f | sort
```

生成が終わったら、出力フォルダと開き方をユーザーに伝える。`index.html` のパスはクリックで開けるように相対パスで示す。
ユーザーは `index.html` をブラウザで開くだけで、ネット接続なしに全仕様書を閲覧できる（`open docs-site/index.html` でも開ける）。

以下を簡潔に報告する。

```
【HTML変換完了】
- 変換したファイル: 3件（index / 基本設計書 / 詳細設計書）
- 出力先: docs-site/
- 開き方: docs-site/index.html をブラウザで開く（ネット接続不要）
```

## オフラインで開ける仕組み

- Mermaid.js はスキルに同梱した `assets/mermaid.min.js`（`globalThis.mermaid` を露出する古典的スクリプト）を使う。
  ES Module ではないため `file://` で開いても CORS エラーにならない。
- CSS は各 HTML に直接埋め込まれるので、スタイル用の外部ファイルは不要。
- 既定構成では出力フォルダ内の `assets/mermaid.min.js` だけが共有ファイル。**出力フォルダごと**移動すれば動く。

> [!IMPORTANT]
> 既定構成では、単一の HTML ファイルだけを取り出して別の場所に置くと Mermaid が描画されない
> （`assets/` への相対パスが切れるため）。フォルダごと扱うか、`--standalone` を使うこと。

## トラブルシューティング

| 症状 | 原因と対処 |
|------|-----------|
| Mermaid図が描画されずコードのまま表示される | フェンスの言語指定が `mermaid` になっているか確認。図の構文エラーの可能性もある |
| リンクをクリックしても飛ばない | 入力に対応する `.md` が存在しない（張り替え先の `.html` が生成されていない）。参照先のファイルも一緒に変換する |
| 図が表示されない（単一ファイルを移動した） | `assets/mermaid.min.js` への相対パス切れ。`--standalone` で再生成するか、フォルダごと配置する |
| 日本語ファイル名のHTMLが開けない | ブラウザ側の問題。ファイル名はそのままで問題ないが、必要ならユーザーに英数字名を提案する |

## 補足: system-spec-writer との連携

このスキルは `system-spec-writer` の出力を引き継ぐことを主目的にしている。
同一セッションで仕様書を生成した直後なら、生成先の `docs/` をそのまま STEP 2 に渡せる。
仕様書を更新したあとに再度 HTML 化する場合も、同じコマンドを再実行すれば出力が上書きされる。
