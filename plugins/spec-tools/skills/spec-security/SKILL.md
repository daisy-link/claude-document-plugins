# 仕様書ベース セキュリティチェック・スキル

`docs/` の仕様書に定義された認証・認可要件を正とし、実装がそれを満たしているかを確認する。
そのうえで、仕様書だけでは拾えない一般的な脆弱性（OWASP Top 10相当の観点）もあわせてチェックする。

> [!IMPORTANT]
> このスキルは**単体でユーザーから直接呼び出さない**。セキュリティチェックに関する依頼は必ず
> `../spec-dev-cycle/SKILL.md` が窓口となり、そこから本ファイルが読み込まれて実行される。

> [!CAUTION]
> `../共通ルール.md` に spec-tools プラグイン全体の絶対禁止事項とトークン節約方針がある。
> 指摘のサンプルコード等に実在の認証情報・個人情報を書き込まない。修正の繰り返しが必要な
> 場合は最大5回までとする（呼び出し元 spec-dev-cycle の繰り返しルールに従う）。

---

## STEP 1: 前提確認（必須）

```bash
find docs/ -name "*.md" 2>/dev/null | sort
```

- `docs/` に仕様書が見つからない場合 → チェックを開始せず、次のように案内して停止する：

  > 「`docs/` に仕様書が見つかりません。仕様書に基づく認証・認可チェックには仕様書が必要です。
  > 先に architecture-design スキルで仕様書を作成しますか？」

---

## STEP 2: 認証・認可要件の抽出

以下から、画面・API・機能ごとの認証要否とロール／権限要件を抽出する。**直接関連する箇所だけを
参照し、`docs/` 配下を無差別に全件読み込まない**（`../共通ルール.md` のトークン節約方針）。

| 仕様書 | 抽出する内容 |
|--------|------------|
| 要件定義書 | 想定されるユーザー種別・ロール、非機能要件のセキュリティ関連記述 |
| システム構成書 | 認証方式（セッション/JWT/OAuth等）、外部IdP連携の有無 |
| 画面設計書 | 画面ごとのログイン要否、アクセス可能なロール |
| 詳細設計（API等） | エンドポイントごとの認証要否、必要な権限、入力パラメータのバリデーション仕様 |

抽出結果を要件一覧として整理する（例：「管理画面配下は管理者ロールのみアクセス可」
「/api/orders は要ログイン、自分の注文のみ参照可」）。

---

## STEP 3: 実装との突き合わせ（仕様適合チェック）

抽出した要件と実装コードを突き合わせ、以下を確認する：

- 認証が必要な画面・APIに、認証チェック（ミドルウェア・デコレータ等）が実装されているか
- ロール・権限チェックが仕様どおりか（例：管理者専用のはずが一般ユーザーでもアクセスできる）
- 「自分のデータのみ参照可」等の**所有者チェック**が抜けていないか（他人のIDを指定して
  データを取得・更新できてしまう、いわゆるIDOR/水平権限昇格が無いか）
- 未ログイン状態からの直接アクセスで意図しない情報が返っていないか

---

## STEP 4: 一般的な脆弱性チェック

仕様書に明記が無くても、実装コードから以下の観点を確認する：

| カテゴリ | チェック観点 |
|---------|------------|
| 入力値検証 | 仕様書のバリデーション仕様どおりに実装され、かつ未検証の入力をそのまま使っていないか |
| インジェクション | SQL/コマンド/テンプレートインジェクションの可能性（文字列結合によるクエリ生成等） |
| 認証情報の扱い | パスワードの平文保存・ログ出力、トークンのクライアント露出、APIキーのハードコード |
| セッション/トークン | 有効期限、再発行・失効の仕組み、CSRF対策の有無 |
| 出力エンコーディング | XSS対策（テンプレートエンジンのエスケープ有効化状況等） |
| エラーハンドリング | エラーメッセージでの内部情報（スタックトレース・SQL・パス）の露出 |
| 依存パッケージ | 既知の脆弱性がある古いバージョンの利用（`package.json` 等のバージョンを確認） |

---

## STEP 5: 結果の報告

重要度別に整理して報告する。

| 重要度 | 内容 |
|--------|------|
| 🔴 Critical | 認証・認可バイパスなど、直接的なデータ漏洩/改ざんにつながるもの |
| 🟠 High | 特定条件下で悪用可能な脆弱性 |
| 🟡 Medium | 直接の脆弱性ではないが防御が手薄な箇所 |
| 🔵 Info | ベストプラクティスからの逸脱（緊急性は低い） |

```
【セキュリティチェック結果】
対象: src/routes/orders.ts（仕様書: docs/詳細設計/API_注文管理.md）

🔴 Critical (1件)
- GET /api/orders/:id で注文IDのみを検証し、リクエストユーザーが注文の所有者かを
  チェックしていない（仕様書「自分の注文のみ参照可」に違反、IDOR）
  → orders.ts:28

🟡 Medium (1件)
- パスワードリセットトークンの有効期限が実装されていない（仕様書に期限の記載なし。
  一般的なベストプラクティスとして推奨）

問題なしの場合:「仕様書に基づく認証・認可チェック、および一般的な脆弱性チェックを実施。
検出事項はありませんでした。」
```

- 修正は自動で行わず、まず結果を報告する。ユーザーが修正を依頼した場合のみ修正する。
- 認証・認可要件の記載が薄い程度であれば Medium〜Info として指摘し報告に含めればよい。ただし、
  **認証・認可要件がまったく読み取れず、チェック自体の前提が成立しない**場合は、報告を待たず
  `../spec-dev-cycle/SKILL.md` の「仕様不明・コーディング規約不在時の共通確認フロー」に従って
  その場でユーザーに確認する（本スキル自身は `docs/` を書き換えない。反映は architecture-design
  経由で行う）。

## 品質チェックリスト

- [ ] チェック開始前に `docs/` の仕様書の有無を確認し、無ければ実施せず案内している
- [ ] 仕様書から認証・認可要件を抽出してから実装と突き合わせている
- [ ] 所有者チェック漏れ（IDOR/水平権限昇格）を確認している
- [ ] 仕様書に無い一般的な脆弱性（インジェクション・XSS・機密情報露出等）も確認している
- [ ] 重要度（Critical/High/Medium/Info）を分けて報告している
- [ ] 破壊的な確認（実際に攻撃リクエストを送る等）を行わず、コードの静的な確認に留めている
- [ ] 直接関連する仕様書だけを読み込んでいる（`docs/` を無差別に全件読み込んでいない）
- [ ] 指摘・修正の記載に実在の認証情報・個人情報を書き込んでいない
