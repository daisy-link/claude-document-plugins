# 受け入れテスト項目書作成スキル

**クライアント（発注者・事業側の担当者）が実際に操作して検証できる**受け入れテスト項目書を作成する。
読者は技術者ではない前提で、平易な言葉で書く。

> [!NOTE]
> **実装完了後だけでなく、開発前でも作成できる。** `docs/画面設計書/` と `docs/要件定義書.md` さえ
> 存在すれば動く（architecture-design の開発前モードで作った、まだ実装が無い設計書ベースでもよい）。
> 開発前モードの設計書から作成した場合は、STEP 3 の手順に従い「想定ベース」である旨を明記する。

> [!IMPORTANT]
> このスキルは**独立して直接呼び出す**。`spec-dev-cycle`（実装・レビュー・テスト・セキュリティ・
> パフォーマンスの窓口）の内部スキルではない。ただし `spec-dev-cycle` は開発サイクル完了時に
> このスキルの利用を提案してよい（`../spec-dev-cycle/SKILL.md` STEP 6 参照）。

> [!CAUTION]
> `../共通ルール.md` に spec-tools プラグイン全体の絶対禁止事項がある。テスト項目書に実在の
> 個人情報・機密情報を書き込まない。

## 対象と対象外

| 対象 | 対象外 |
|------|--------|
| `docs/画面設計書/` の各画面：クライアントが画面を操作して確認できる内容 | API単体・バッチ・RPA・外部連携・共通処理（`docs/詳細設計/`）。クライアントが直接操作・観測できないため、開発者向けの自動テスト（`spec-test`）でカバーする |
| `docs/要件定義書.md` の機能要件：発注した機能が実現されているかの確認 | 非機能要件（性能・可用性等）の数値検証。専門的な計測が必要なため対象外 |

## STEP 1: 前提確認

```bash
find docs/ -name "*.md" 2>/dev/null | sort
find docs/画面設計書 -name "*.md" 2>/dev/null | sort
```

- `docs/` に仕様書が見つからない場合 → 作成せず、次のように案内して停止する：

  > 「`docs/` に仕様書が見つかりません。受け入れテスト項目書の作成には仕様書が必要です。
  > 先に architecture-design スキルで仕様書を作成しますか？」

- `docs/画面設計書/` が無い（画面の無いシステム）場合 → 画面ベースの項目は作れない旨をユーザーに伝え、
  `docs/要件定義書.md` の機能要件だけで作成してよいか確認する。

### 実装が完了しているかの確認（開発前でも作成できる）

このスキルはコードを読まず `docs/画面設計書/` と `docs/要件定義書.md` だけを情報源にするため、
**実装がまだ無くても動く。** ただし、実装の有無によって文書に添える注記が変わるので確認する。

```bash
# 画面設計書に記載のルートに対応する実装ファイルがあるか、ざっと確認する
find . -type f -not -path './.git/*' -not -path './docs/*' -not -path './node_modules/*' | head -50
```

- 画面設計書のルートに対応するアプリケーションコードが無い、または画面設計書・要件定義書の
  見出しに「（予定）」等、architecture-design の開発前モードで作られた形跡がある場合 →
  **「開発前モードの設計書ベース」** として扱う。STEP 3 で作成する文書に、以下の注記を追加する：

  ```markdown
  > [!NOTE]
  > 本書は設計段階の内容にもとづく想定のテスト項目です。実装完了後、実際の画面と
  > 内容が異なる場合は見直しが必要です。
  ```

- 実装が存在する（通常のケース）→ 注記は付けず、STEP 3 の記載例どおりに作成する。

## STEP 2: 新規作成 or 更新の判定

| 判定条件 | モード |
|---------|--------|
| `docs/受け入れテスト項目書.md` が存在しない | **新規作成モード** → STEP 3へ |
| 存在する（画面・機能が追加された等で更新したい） | **更新モード** → STEP 5へ |

## STEP 3: 項目の洗い出し（新規作成）

1. `docs/画面設計書/index.md` と各 `P0X_画面名.md` を読み込み、画面ごとに
   「クライアントが実際に行う操作」と「その結果どうなればよいか」を1〜数項目にまとめる。
   画面のルート（URL/URI）を確認内容に含める
2. `docs/要件定義書.md` の機能要件一覧（4.1 主要機能一覧）を読み込み、各機能について
   「その機能が実際に使えるか」を確認する項目を1行にまとめる
3. 技術的な表現（API名・テーブル名・ステータスコード等）は使わず、業務的な言葉で書く

### 記載する列

| No | 画面・機能名 | 確認内容 | 結果 | 確認日 | 担当者 | 備考 |
|----|------------|---------|------|--------|--------|------|

- **確認内容**: 手順と期待結果をまとめた平易な文章（例：「`/login` にアクセスし、正しいID・
  パスワードでログインできること」）。画面系の項目にはURLを含める
- **結果**: `[ ]` = 未実施、`[x]` = OK。**新規作成時はすべて `[ ]`**（NGの場合はチェックを
  入れず備考に内容を書く運用）
- **確認日・担当者**: 新規作成時は空欄（クライアント側が実施時に記入する）

### 記載例

```markdown
| No | 画面・機能名 | 確認内容 | 結果 | 確認日 | 担当者 | 備考 |
|----|------------|---------|------|--------|--------|------|
| 1 | ログイン画面 | `/login` にアクセスし、正しいID・パスワードでログインできること | [ ] | | | |
| 2 | ログイン画面 | 誤ったパスワードでログインしようとするとエラーメッセージが表示されること | [ ] | | | |
| 3 | 商品管理機能 | 商品の登録・編集・削除が一通り行えること（要件定義書 F03） | [ ] | | | |
```

> [!NOTE]
> 表のセル内の `[ ]` / `[x]` はクリックで切り替わる本物のチェックボックスにはならない
> （表のセル内では文字としてしか扱われない）。確認したらセル内のスペースを `x` に書き換える運用にする。

## STEP 4: 検収欄の追加

一覧表の後に、検収の署名欄を追加する。

```markdown
---

## 検収

上記の内容を確認し、本システムを検収します。

| 項目 | 内容 |
|------|------|
| 会社名 | |
| ご担当者名 | |
| 確認日 | |
```

## STEP 5: 保存・更新

出力先は `docs/受け入れテスト項目書.md`（システム名を含めない、資料名のみ）。

**新規作成の場合**：STEP 3・4の内容でファイルを作成する。`docs/index.md` が存在すれば、
ドキュメント一覧に1行追加する（無ければ追加しなくてよい）。

**更新モードの場合**（すでにファイルが存在する）：

1. `docs/画面設計書/` と `docs/要件定義書.md` を読み込み、前回作成時から**新しく増えた画面・機能**を検出する
2. 新しく増えた項目だけを、一覧表の末尾に追加する（No は既存の最大値の続きから）。結果は `[ ]`
3. **既存の行（結果・確認日・担当者・備考を含む）は一切書き換えない。** クライアントが既に
   記入・検収した内容を上書きしない
4. 画面・機能が削除された場合、行は削除せず残し、備考に「削除された機能のため対象外」等と
   追記する（検収の証跡を残すため。他の仕様書のように行ごと削除しない）
5. 更新した旨と追加件数を報告する

```
【受け入れテスト項目書 更新完了】
- 出力先: docs/受け入れテスト項目書.md
- 追加した項目: 2件（No.15〜16、注文管理機能の追加に伴う）
- 既存の項目・記入済みの結果: 変更なし（14件のまま）
```

## STEP 6: Excel出力（希望があった場合のみ）

**`docs/受け入れテスト項目書.md` が正であり、Excelはそこから作る追加のオプション出力。**
ユーザーから明示的に要望されたときだけ行う（自動では作らない）。1シートに全項目を詰め込むと
画面数が多いプロジェクトで膨大になるため、**業務ドメイン単位でシートを分割する。**

1. `docs/受け入れテスト項目書.md` の最新内容を読み込む
2. 画面系の項目を、ルートのパス階層（例：`/member/*` → 会員関連、`/orders/*` → 注文関連、
   `/admin/*` → 管理画面）と画面名・確認内容から**業務ドメインごとにグルーピングする**。
   はっきり分類できない画面は「その他」としてまとめる。ドメイン名・グルーピングは
   プロジェクトに応じて判断してよい（機械的な分割ではなく、クライアントが見て分かりやすい単位を優先する）
3. `xlsx` スキルを使って `docs/受け入れテスト項目書.xlsx` を生成する。シート構成：
   - **画面系のシート（ドメインごとに1枚）**: 例）「会員関連」「注文関連」「管理画面」「その他」。
     列は No / 画面名 / 確認内容 / 結果 / 確認日 / 担当者 / 備考
   - **シート「機能確認」**: `docs/要件定義書.md` 由来の機能単位の項目をまとめる。
     列は No / 機能名 / 確認内容 / 結果 / 確認日 / 担当者 / 備考
   - **シート「検収」**: 会社名・ご担当者名・確認日の記入欄
   - 全シート共通で、ヘッダー行は太字・背景色で見やすくする。**結果列は `未実施` / `OK` / `NG`
     から選べるドロップダウン（データ入力規則）にする**（Markdown側の `[ ]` は「未実施」、
     `[x]` は「OK」に変換する。NGだった行は「NG」にし、備考の内容もそのまま引き継ぐ）
   - Excelのシート名は31文字以内、`: \ / ? * [ ]` を含めない
4. Markdown側のNo番号は変えず、Excel側の各シートにもそのNo番号をそのまま引き継ぐ
   （シートをまたいでも同じ項目だと分かるように）
5. 生成後、どのようにシートを分けたか（ドメイン名と件数）を完了報告に含める。グルーピングが
   意図と違う場合はユーザーに直してもらえるようにする
6. `docs/受け入れテスト項目書.md` を更新したあとに再度Excel化を頼まれたら、その時点の
   Markdownの内容で作り直す（Excel側で加筆された結果・備考は保持されない前提をユーザーに伝える。
   Excel側の記入結果をMarkdownに反映したい場合は、その内容を教えてもらい手動で反映する）

```
【Excel出力完了】
- 出力先: docs/受け入れテスト項目書.xlsx
- シート構成:
  - 会員関連（8件）
  - 注文関連（12件）
  - 管理画面（5件）
  - その他（3件）
  - 機能確認（10件）
  - 検収
```
