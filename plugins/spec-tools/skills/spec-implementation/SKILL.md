  コーディング規約が仕様書に含まれていない場合や、仕様が不明で判断がつかない場合は自動で方針を
  決めず、spec-dev-cycle の共通確認フローに従って作業を中断する。

# 仕様書ベース実装スキル

`docs/` の仕様書を正として、そこに書かれた仕様どおりにソースコードを実装する。
仕様書に書かれていない挙動を勝手に補ってはいけない。曖昧な箇所はユーザーに確認する。

> [!IMPORTANT]
> このスキルは**単体でユーザーから直接呼び出さない**。実装に関する依頼は必ず
> `../spec-dev-cycle/SKILL.md` が窓口となり、そこから本ファイルが読み込まれて実行される。

> [!IMPORTANT]
> このスキルは**仕様書が無いと動作しない**。`docs/` が存在しない、または対象機能の仕様書が
> 見つからない場合は実装に着手せず、architecture-design スキルでの仕様書作成を先に案内する。

> [!CAUTION]
> `../共通ルール.md` に spec-tools プラグイン全体の絶対禁止事項（機密情報の書き込み禁止・
> 繰り返し処理の上限・環境起因エラーのカウント除外・常駐プロセスの起動禁止）とトークン節約
> 方針がある。STEP 5のレビュー・修正ループの「最大5回」はこの共通ルールに基づき、このループの
> 修正回数は呼び出し元 spec-dev-cycle が管理する**サイクル全体の修正上限（合計10回）**にも
> 算入される。

---

## STEP 1: 前提確認（必須）

### 1-1. 仕様書の有無を確認する

```bash
find docs/ -name "*.md" 2>/dev/null | sort
```

- `docs/` が存在しない、または関連する仕様書が見つからない場合 → 実装を開始せず、次のように案内して停止する：

  > 「`docs/` に仕様書が見つかりません。実装の前提として仕様書が必要です。
  > 先に architecture-design スキルで仕様書を作成しますか？」

- 仕様書はあるが実装対象の機能に関する記載が無い場合 → その旨をユーザーに伝え、対象範囲を確認する。

### 1-2. モックの有無を確認する（任意）

画面のスクリーンショット・ワイヤーフレーム・Figma書き出し画像・デザインファイルなどが
アップロードされていれば、画面レイアウト・配色・文言の参考として使う。**無くても実装は進める。**

### 1-3. コーディング規約の有無を確認する

```bash
test -f docs/コーディング規約.md && echo "コーディング規約あり" || echo "コーディング規約なし"
```

- `docs/コーディング規約.md` がある場合 → 必ず読み込み、実装全体を通してそのルールに従う。
- 無い場合 → **自動で方針を決めない。** `../spec-dev-cycle/SKILL.md` の「仕様不明・コーディング
  規約不在時の共通確認フロー」に従い、実装に着手する前に処理を中断してユーザーに方針を確認する。

---

## STEP 2: 実装対象の特定と仕様書の読み込み

1. ユーザーの依頼から実装対象（画面・API・バッチ・機能単位）を特定する
2. 対象に**直接関連する**仕様書を読み込む。`docs/` 配下を無差別に全件読み込まず、以下の
   対応表を目安に必要な範囲だけに絞る（`../共通ルール.md` のトークン節約方針）：

| 実装対象 | 読み込む仕様書 |
|---------|--------------|
| 画面 | `docs/画面設計書/` 内の該当画面ファイル、関連する `docs/DB設計.md`、関連する詳細設計（呼び出すAPI等） |
| API・バッチ・外部連携等 | `docs/詳細設計/` 内の該当ファイル、関連する `docs/DB設計.md` |
| DB変更を伴うもの | `docs/DB設計.md` の該当テーブル定義 |
| 全体像の把握 | `docs/要件定義書.md`、`docs/システム構成書.md`、`docs/index.md` |

3. 仕様書内に `[要確認]` の記載や、仕様書に書かれていない挙動が必要な箇所を見つけたら、
   その場で仕様を決めてはいけない。読み込みが終わった時点で不明点を**一括して**洗い出し、
   `../spec-dev-cycle/SKILL.md` の「仕様不明・コーディング規約不在時の共通確認フロー」
   （5-0 事前一括チェック）に従い、実装着手前にまとめてユーザーに確認する
   （実装中に1件ずつ中断を繰り返さない）。

---

## STEP 3: 実装方針の提示（規模が大きい場合）

対象が複数ファイル・複数機能にまたがる場合は、いきなり実装せずタスクを分解して提示し、
ユーザーの合意を得てから着手する。

```
【実装方針】
仕様書: docs/詳細設計/API_商品管理.md

1. GET /api/items 一覧取得エンドポイントの追加
2. items テーブルへの stock カラム追加（マイグレーション）
3. 在庫チェックのバリデーション追加

この順番で進めてよいですか？
```

小規模な変更（1ファイル・1機能程度）の場合はこのステップを省略し、直接実装してよい。

---

## STEP 4: 実装

- 仕様書に書かれた項目（画面項目・入力チェック・API入出力・エラーレスポンス・DBカラム等）を
  漏れなく実装する。
- モックがある場合は見た目をモックに合わせるが、挙動・データ構造は仕様書を正とする
  （モックと仕様書が矛盾する場合は仕様書を優先し、矛盾をユーザーに報告する）。
- 既存プロジェクトの場合、既存コードの書き方（フレームワークの使い方・ディレクトリ構成・
  命名規則）に合わせる。新規プロジェクトの場合、`docs/システム構成書.md` や
  `docs/開発環境セットアップ.md` に記載の技術スタックに従う。
- 仕様書に無い機能・項目を勝手に追加しない。あった方が良いと思う場合は実装後に提案として伝える。
- **gitなしモード**（spec-dev-cycle が git 未導入と判定し、ユーザーが git なしを選択した場合）
  では、既存ファイルを変更・削除する前にバックアップディレクトリへコピーし、新規作成した
  ファイルを一覧として記録する（手順は `../spec-dev-cycle/SKILL.md` STEP 2 を参照）。

> [!NOTE]
> 認証情報・APIキー・接続文字列などの機密値は仕様書上マスキングされている。実装時は
> `.env.example` 等のプレースホルダーを使い、実際の値をコードやコミットに含めない。

### 実装後の静的チェック（lint・型チェック・ビルド）

コードを書き終えたら、STEP 5のレビューに進む前に、プロジェクトに設定されている
lint・型チェック・ビルドを実行してエラーが無いことを確認する
（例：`npm run lint`、`npx tsc --noEmit`、`npm run build`。プロジェクトの `package.json` や
CI設定にあるコマンドを使う。設定が無い場合はスキップしてよい）。

- エラーがあれば修正してからSTEP 5に進む（コンパイルが通らないコードをレビューに回さない）
- このエラー修正の繰り返しも最大5回まで（`../共通ルール.md`）。解消しない場合は中止して報告する
- コマンドはワンショット実行し、watch モードを使わない（`../共通ルール.md`）

---

## STEP 5: 実装後レビュー（必須・省略不可・最大5回）

実装が終わったら、**必ず** `../spec-review/SKILL.md` の手順でレビューを行う。レビューを
省略してSTEP 6の完了報告に進んではいけない。**レビュー実行は最大5回までとし、それ以上は
繰り返さない**（`../共通ルール.md` の絶対禁止事項「繰り返し処理は最大5回まで」）。また、
このループでの修正回数は spec-dev-cycle が管理するサイクル全体の修正上限（合計10回）に
算入され、全体上限に達した場合はレビュー回数が残っていても中止する。

1. `../spec-review/SKILL.md` の手順に従い、STEP 4で実装したコードをレビューする（1回目）
2. 🔴違反が見つかった場合 → STEP 4に戻って修正し、再度レビューする
   - **1回目〜3回目のレビューで見つかった🔴違反は自動で修正する（確認不要）**。修正後、
     再度レビューを行う（3回目のレビューNGまでは、確認を挟まず自動で修正・再レビューを
     繰り返してよい）。
   - **4回目のレビューでも🔴違反が残る場合は、ユーザーに確認する**：

     ```
     【レビュー結果】4回目もNGでした
     🔴 GET /api/items のレスポンスに stock フィールドが含まれていない（再掲）

     このまま自動でもう1回修正しますか？それとも一旦ここで止めますか？
     ```

     - 承認を得たら**1回だけ**追加で修正・再レビューする
     - 中止の指示があれば、その時点で実装を止めSTEP 6の報告に進む（残っている🔴違反は未解消として報告する）
3. **5回目のレビューでも🔴違反が残る場合は、ユーザーの意向にかかわらずそこで処理を中止する**
   確認を挟まず「レビューが5回連続でNGのため、これ以上は繰り返さず中止します」と報告し、
   残っている🔴違反を明記してSTEP 6の報告に進む
4. 🟡要確認・🔵規約違反は自動修正の対象にしない。STEP 6の報告にまとめて記載し、対応するかはユーザーに委ねる
5. 🔴違反が無い、または解消できた場合はSTEP 6に進む

> [!CAUTION]
> 4回目のレビューNGに対する修正は、ユーザーの明示的な承認なしに自動実行しない。
> **レビュー実行が5回に達したら、ユーザーが続行を希望しても打ち切る。**

---

## STEP 6: 仕様書との整合性確認と報告

実装後、書いたコードを仕様書と見比べて漏れがないか確認し、STEP 5のレビュー結果とあわせて報告する。

```
【実装完了】
仕様書: docs/詳細設計/API_商品管理.md

- 実装したファイル:
  - src/routes/items.ts（GET /api/items を追加）
  - src/db/migrations/xxx_add_stock.sql（items.stock カラム追加）
- 静的チェック: lint・型チェック・ビルドすべて通過
- レビュー: 🔴違反1件（stockフィールド欠落）を1回目の自動修正で解消。🟡要確認1件は未対応（在庫0件時の仕様が未確定）
- 仕様書との差異: なし
- [要確認]だった項目: 在庫0件時のレスポンス仕様が未確定のため、404ではなく空配列を返す実装にしました。問題なければこのままにします。
```

差異や仕様書側の記載漏れに気づいた場合は実装を止めずに報告し、必要ならユーザーに仕様書側の
修正も提案する（このスキル自身は `docs/` の仕様書ファイルを書き換えない）。

> [!TIP]
> テスト作成・セキュリティ/パフォーマンスチェックは、それぞれ spec-test・spec-security・
> spec-performance スキルが担当する。これらの呼び出しも spec-dev-cycle が窓口になるため、
> 必要に応じてユーザーに「続けてテストを作成しますか？」等を spec-dev-cycle 側から案内する。
> 実装後のレビューは本スキルに組み込まれているため、改めて spec-review を呼び出す必要はない。

## 品質チェックリスト

- [ ] 実装前に `docs/` の仕様書の有無を確認し、無ければ実装せず案内している
- [ ] 対象範囲の仕様書をすべて読み込んでから実装している
- [ ] `docs/コーディング規約.md` があればそれに従っている
- [ ] `docs/コーディング規約.md` が無い場合、自動で方針を決めず spec-dev-cycle の共通確認フローに従っている
- [ ] モックは見た目の参考にとどめ、挙動は仕様書を優先している
- [ ] 仕様書に無い機能を勝手に追加していない
- [ ] 仕様書内の `[要確認]` や仕様不明点は、自動で決めず spec-dev-cycle の共通確認フローに従っている
- [ ] 不明点は実装着手前に一括で確認し、実装中に1件ずつ中断を繰り返していない
- [ ] 規模が大きい場合、実装前にタスク分解を提示し合意を得ている
- [ ] 機密情報（APIキー・接続文字列等）をコードに直書きしていない
- [ ] レビュー前に lint・型チェック・ビルドを実行し、エラーを解消している（プロジェクトに設定がある場合）
- [ ] 実装後、必ず spec-review の手順でレビューを実施している（省略していない）
- [ ] レビューで🔴違反が出た場合、3回目までは自動修正、4回目はユーザーに確認している
- [ ] レビューが5回目もNGの場合、ユーザー確認なしで処理を中止している（無制限に繰り返していない）
- [ ] レビュー結果（違反・要確認・規約違反）を最終報告に含めている
- [ ] 生成・編集したコードに機密情報・個人情報を書き込んでいない（`../共通ルール.md`）
- [ ] 実装後、仕様書との差異の有無を報告している
