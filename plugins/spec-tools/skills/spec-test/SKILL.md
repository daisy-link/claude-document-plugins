# 仕様書ベース テスト作成スキル

`docs/` の仕様書に書かれた挙動を正として、テストケースを網羅的に洗い出し、テストコードを作成する。
仕様書に無い挙動を勝手に「正しい仕様」として想定してテストを書かない。

> [!IMPORTANT]
> このスキルは**単体でユーザーから直接呼び出さない**。テスト作成に関する依頼は必ず
> `../spec-dev-cycle/SKILL.md` が窓口となり、そこから本ファイルが読み込まれて実行される。

> [!CAUTION]
> `../共通ルール.md` に spec-tools プラグイン全体の絶対禁止事項とトークン節約方針がある。
> テストのフィクスチャ・ダミーデータに実在の個人情報を書き込まない。テストの再修正が
> 必要な場合は最大5回までとする（呼び出し元 spec-dev-cycle の繰り返しルールに従う）。
> **テストは必ずワンショット形式で実行する（watchモード禁止・タイムアウト付き）。**

---

## STEP 1: 前提確認（必須）

```bash
find docs/ -name "*.md" 2>/dev/null | sort
```

- `docs/` に仕様書が見つからない場合 → テスト作成を開始せず、次のように案内して停止する：

  > 「`docs/` に仕様書が見つかりません。仕様ベースのテスト作成には仕様書が必要です。
  > 先に architecture-design スキルで仕様書を作成しますか？」

### 既存のテスト環境を把握する

```bash
# テストフレームワーク・既存テストの有無を確認
find . -not -path './.git/*' -not -path './node_modules/*' -not -path './docs/*' \
  \( -iname "*test*" -o -iname "*spec*" \) -type f | sort | head -50
cat package.json 2>/dev/null | grep -A5 '"scripts"'
```

既存テストがあれば、そのフレームワーク（Jest/Vitest/pytest/RSpec/JUnit等）・命名規則・
ディレクトリ構成・モックの流儀に合わせる。既存テストが無い場合は、`docs/開発環境セットアップ.md`
や `docs/システム構成書.md` に記載の技術スタックに適したフレームワークを提案してから作成する。

---

## STEP 2: テスト対象の特定と仕様書の読み込み

ユーザーの依頼から対象（画面・API・バッチ・関数単位）を特定し、**直接関連する**仕様書だけを
読み込む（`../共通ルール.md` のトークン節約方針。`docs/` を無差別に全件読み込まない）。

| テスト対象 | 読み込む仕様書 |
|-----------|--------------|
| 画面 | 該当する `docs/画面設計書/` のファイル（入力チェック・画面遷移条件） |
| API・バッチ等 | 該当する `docs/詳細設計/` のファイル（入出力・エラーレスポンス） |
| DB絡みの処理 | `docs/DB設計.md` の制約（NOT NULL・UNIQUE・外部キー等） |

---

## STEP 3: テストケースの洗い出し

仕様書の記載から、以下の観点でテストケースを洗い出す：

| 観点 | 内容 |
|------|------|
| 正常系 | 仕様どおりの入力での期待される出力・挙動 |
| 異常系 | 仕様書に定義されたエラーケース（バリデーションエラー・認可エラー・存在しないリソース等） |
| 境界値 | 文字数上限、数値の最小/最大、必須/任意項目の有無切り替わり |
| 状態遷移 | 画面遷移条件、ステータスの変化（仕様書に記載がある場合） |

洗い出したテストケースは、コードを書く前に一覧としてユーザーに提示する：

```
【テストケース一覧】
対象: docs/詳細設計/API_商品管理.md（POST /api/items）

1. 正常系: 必須項目をすべて指定 → 201 で作成される
2. 異常系: name が未指定 → 400 バリデーションエラー
3. 異常系: price が負の数 → 400 バリデーションエラー
4. 境界値: name が仕様上限の100文字ちょうど → 201 で作成される
5. 境界値: name が101文字 → 400 バリデーションエラー

このケースでテストコードを作成してよいですか？追加・削除があれば教えてください。
```

ケース数が少ない（3件以下程度）場合は確認を省略し、直接テストコードの作成に進んでよい。

---

## STEP 4: テストコードの作成

- 既存プロジェクトの場合、既存テストと同じフレームワーク・アサーションスタイル・
  ファイル配置規則に合わせる。
- テスト名・description は、何を検証しているかが仕様書と対応付けて分かるように書く
  （例：`名前が101文字のとき400を返す`）。
- 外部依存（DB・外部API等）は既存テストのモック/フィクスチャの流儀に合わせる。
  流儀が無い新規プロジェクトの場合は言語・フレームワークの一般的な手法を使う。
- 仕様書に無いケースを推測で追加しない。軽微な不足（追加を推奨するケースがある程度）は
  テスト作成後に提案として伝えればよいが、**期待値そのものが仕様書から読み取れず判断がつかない**
  場合は、推測でテストを書かず `../spec-dev-cycle/SKILL.md` の「仕様不明・コーディング規約不在時の
  共通確認フロー」に従ってユーザーに確認する。
- テストを実行する場合は必ず**ワンショット形式**で実行する（watch モード禁止。`vitest run`・
  `jest --watchAll=false`・`CI=true npm test` 等、1回で終了するコマンドを使う）。タイムアウトを
  設定し、開発サーバー等の常駐プロセスを起動しない（`../共通ルール.md`）。
- テスト実行が環境起因（依存パッケージ未インストール・DB未起動・設定不足等）で失敗した場合は、
  修正ループを回さず即座にユーザーへ報告する（`../共通ルール.md`）。
- 新規テストの実行に合わせて、**既存のテストスイート（少なくとも関連するスイート）も実行**し、
  今回の変更による回帰が無いことを確認する（回帰があれば失敗として報告する）。

---

## STEP 5: 結果の報告

```
【テスト作成完了】
仕様書: docs/詳細設計/API_商品管理.md

- 作成したファイル: tests/routes/items.test.ts（5ケース）
- 実行コマンド: npm test -- items.test.ts
- 仕様書ではカバーされていないが追加を推奨するケース:
  - 同時リクエストによる重複作成（仕様書に排他制御の記載が無いため未実装）
```

## 品質チェックリスト

- [ ] テスト作成前に `docs/` の仕様書の有無を確認し、無ければ実施せず案内している
- [ ] 既存のテストフレームワーク・構成があればそれに合わせている
- [ ] 正常系・異常系・境界値を仕様書の記載に基づいて洗い出している
- [ ] ケース数が多い場合、コード作成前に一覧を提示し合意を得ている
- [ ] 仕様書に無い挙動を推測でテストに書いていない
- [ ] 直接関連する仕様書だけを読み込んでいる（`docs/` を無差別に全件読み込んでいない）
- [ ] テストのフィクスチャ・ダミーデータに実在の個人情報を使っていない
- [ ] テストをワンショット形式で実行・案内している（watchモード・常駐プロセスを起動していない）
- [ ] 環境起因の実行失敗を修正ループに含めず、即座に報告している
- [ ] 既存テストも実行し、回帰が無いことを確認している
- [ ] 作成後、実行コマンドを案内している
