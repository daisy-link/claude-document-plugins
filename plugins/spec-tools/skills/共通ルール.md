# spec-tools 共通ルール

spec-tools プラグインの全スキル（architecture-design・spec-dev-cycle・spec-implementation・
spec-review・spec-test・spec-security・spec-performance・acceptance-test-writer・
spec-alignment-audit）に共通して適用される、絶対禁止事項と運用方針をまとめる。各スキルの
`SKILL.md` はこのファイルを参照し、詳細手順内で個別に矛盾する記述をしない。

> [!NOTE]
> このファイル自体は独立したスキルではない（`name` フロントマターを持たず、ユーザーから
> 直接トリガーされない）。各スキルの `SKILL.md` から相対パス `../共通ルール.md` で読み込まれる。

---

## 絶対禁止事項

### 1. 機密情報・個人情報の書き込み禁止

**生成・編集するすべてのファイル**（仕様書・実装コード・テストコード・設定ファイル等、種類を
問わない）に、以下を書き込むことを禁止する。インプットのファイルに含まれていても出力には
含めない。

| 種別 | 具体例 |
|------|--------|
| 認証情報 | パスワード、APIキー、アクセストークン、シークレットキー、秘密鍵 |
| 接続情報 | DB接続文字列（ユーザー名・パスワード含むもの）、内部IPアドレス、内部ホスト名 |
| 個人情報 | 氏名、メールアドレス、電話番号、住所、生年月日、マイナンバー等の個人を特定できる情報 |
| その他機密情報 | 本番環境の具体的なURL・エンドポイント、ライセンスキー、証明書の内容 |

- 実装コードで機密値が必要な箇所は `.env.example` 等のプレースホルダーを使い、実際の値を
  コード・コミットに含めない。
- テストコードのフィクスチャ・ダミーデータも実在の個人情報を使わない（`test@example.com` の
  ような明らかなダミー値を使う）。
- 仕様書での記載方法（マスキング等の代替表記）は architecture-design の
  「絶対遵守：記載禁止事項」を参照する。

### 2. 繰り返し処理は最大5回まで

実装↔レビューの修正ループ、テストの再修正ループなど、**同一の問題に対して繰り返し処理を行う
場合は最大5回まで**とする。5回に達してもなお解決しない場合は、ユーザーの意向にかかわらず
その時点で処理を中止し、残っている問題を報告する。**無制限に繰り返し処理を行うことを禁止する。**

- 何を「1回」とカウントするかは、その繰り返しループを定義しているスキルの手順に従う
  （例：spec-implementation の実装後レビューループはレビュー実行回数、spec-dev-cycle の
  テスト再修正ループはテスト実行回数）
- ループ内の自動修正は**3回目のNGまで**（確認不要で修正・再実行してよい）。**4回目のNGは
  ユーザーの承認を得てから**修正し、5回目もNGならユーザーの意向にかかわらず中止する
- 各ループのカウントは独立して行う（例：実装後レビューループとテスト再修正ループは別々に
  最大5回までカウントする）
- ただし多重ループの掛け算で長時間実行になることを防ぐため、**1つの機能に対する1回の
  開発サイクル全体では、コードの修正回数の合計を最大10回まで**とする（全体上限）。
  10回に達したら、各ループの残り回数にかかわらずその時点で処理を中止して報告する
- 上限に達する前でも、ユーザーが中止を選んだ場合はその時点で終了する
- **中止した時点で、それまでに試した修正アプローチ（何をどう直そうとしたか）を報告に残す。**
  ユーザーが同じ問題の修正を再依頼した場合は新しいサイクルとしてカウントし直してよいが、
  前回の報告を踏まえ、失敗した修正アプローチをそのまま繰り返さない

#### カウント対象外：環境起因のエラー

テスト失敗・実行エラーの原因が**コードのバグではなく環境の問題**（依存パッケージ未インストール、
DB・外部サービスの未起動、環境変数・設定の不足、権限不足等）である場合は、修正ループのカウント
対象にしない。コードを修正しても解決しないため、ループを回さず**即座に処理を止めてユーザーに
報告**し、環境側の解消方法を案内する（環境が整ってから再開する）。

### 3. 常駐プロセスの起動禁止

スキルの処理として、**終了しないプロセス（常駐プロセス）を起動することを禁止する。**
処理が永遠に返らなくなり、無制限処理の原因になる。

- テストは必ず**ワンショット実行**する（watch モード禁止）。例：`vitest run`（`vitest` 単体は
  watch になるため不可）、`jest --watchAll=false`、`CI=true npm test` など、1回実行して
  終了する形式を使う
- 開発サーバー・ビルドの watch・デーモン等をスキルの手順として起動しない
  （動作確認にサーバー起動が必要な場合は、起動をユーザーに依頼する）
- テスト・ビルド等のコマンドには適切なタイムアウトを設定し、ハングした場合は打ち切って
  環境起因のエラーとして報告する

### 4. 破壊的コマンドの制限

**プロジェクトフォルダの外に対する削除・移動・破壊的な操作を禁止する。**

- `rm`・`rmdir`・`shred`・`find -delete`・`mv` は、プロジェクトフォルダ内および
  一時ディレクトリ（`/tmp` 等）内のパスに対してのみ使用できる。プロジェクト外のパス
  （上位階層・ホームディレクトリ・システム領域）を対象にしてはならない
- `sudo`・`dd`・`mkfs` 系・`diskutil` の消去系・`shutdown`・`reboot` は場所を問わず使用禁止
- プロジェクト外のファイル操作が必要な場面では、コマンドを実行せずユーザーに手動での
  実行を依頼する

> [!NOTE]
> このルールは指示であると同時に、プラグイン同梱の PreToolUse フック
> （`hooks/check_destructive.py`）によって**強制的にも**ブロックされる。フックに拒否された
> 場合は、同じコマンドを言い換えて再試行せず、ユーザーに状況を報告して指示を仰ぐこと。

## トークン節約方針

### 仕様書は関連するものだけを参照する

`docs/` 配下の仕様書を読み込む際は、**作業対象に直接関連する資料だけを優先して参照する。**
`docs/index.md` や各スキルの「読み込む仕様書」の対応表を使って対象を絞り込み、`docs/` 配下を
無差別に全件読み込むことは避ける。

- 対象が明確な場合（特定の画面・API・機能）は、該当する仕様書ファイルのみを読み込む
- 全体像の把握が必要な場合（要件定義書の参照が必要、複数機能にまたがる等）に**限り**、
  関連する他の資料も参照してよい
- 「システム全体をレビューして」のように対象が広い依頼の場合は、対象範囲を絞ってから
  読み込む（一度に `docs/` 配下の全ファイルを読み込まない）
